# Guardrails AI｜把输入输出验证放进 AI 应用运行时 🛡️

- Source｜来源：Guardrails AI GitHub
- Type｜类型：开源 AI 项目产品化分析 / AI Guardrails / Runtime Validation / LLM Safety
- Link｜链接：https://github.com/guardrails-ai/guardrails

Guardrails AI 是一个用于检查和处理 LLM 输入输出的开源框架。

它关注的问题不是怎样让模型更聪明，而是当模型收到不合适的输入，或生成不符合产品要求的结果时，应用如何在运行过程中发现并采取动作。

AI 输出具有不确定性。即使 Prompt、模型和上下文保持不变，结果也可能出现格式错误、敏感内容、无依据陈述或其他不符合业务规则的情况。

从产品视角看，Guardrails AI 代表的是运行时质量控制：

> 产品不能只告诉模型应该怎么做，还需要定义结果不合格时如何识别、反馈和恢复。

---

## What It Is｜它是什么 🧩

Guardrails AI 提供 Input Guard、Output Guard 和可组合的 Validator。

根据项目当前 GitHub 说明，Guard 可以在内容进入模型前检查输入，也可以在模型返回后验证输出。Validator 针对某类风险或要求进行判断，例如格式匹配、特定内容检测或输出质量检查，并根据失败情况触发异常、过滤或其他处理动作。

项目还通过 Guardrails Hub 组织可复用 Validator，让开发者能够组合不同检查，而不必把所有规则写进一段 Prompt。

Guardrails AI 也支持结构化数据生成，但它更值得观察的产品位置是：在用户、模型和下游系统之间增加一个验证层。

这个验证层并不保证模型永远正确。它只能对已经定义并能够检测的风险进行处理，检测本身也可能出错。

---

## Why It Matters｜为什么值得关注 💡

很多 AI 产品依赖 Prompt 约束模型行为。

团队会要求模型使用固定格式、避免敏感内容、引用来源或遵守特定语气。但 Prompt 是行为引导，不是确定性的执行规则。模型可能忽略要求，也可能在复杂上下文中理解偏差。

当输出只用于阅读时，一次失败可能意味着用户重新生成；当输出会进入数据库、自动化流程或 Agent 工具调用时，错误可能继续传播。

例如，字段格式不正确会破坏后续接口，敏感信息可能被发送到不合适的位置，未经验证的结论可能成为下一步决策依据。

Guardrails 值得关注，是因为它让产品从“期待模型遵守规则”转向“检查结果是否满足规则”。

这种变化并不会消除不确定性，但能让部分失败从不可见变成可处理状态。

## Core Capability｜核心能力 ⚙️

Guardrails AI 的产品结构可以从检查位置、验证规则和失败处理三个方向理解。

### Input Validation｜输入验证

输入检查发生在内容发送给模型之前。

它可以用于识别不符合产品范围的请求、敏感信息或已知风险模式。对于连接外部数据的 Agent，输入不仅来自用户，也可能来自网页、文件和工具返回结果，因此边界不能只放在聊天输入框。

输入验证需要谨慎。规则过严会阻止合理任务，规则过松则无法降低风险。产品还要告诉用户为什么无法继续，而不是只返回一个技术错误。

### Output Validation｜输出验证

输出检查判断模型结果是否满足结构、内容或业务要求。

有些条件可以通过确定性规则验证，例如字段存在、格式匹配和数值范围；有些则需要模型或专门分类器判断，例如语义相关性和有害内容。

不同验证方式的速度、成本和可靠性不同，不能把所有问题都交给另一个模型评分。

### Failure Handling｜失败处理

发现问题后，产品需要决定下一步。

可能的处理包括拒绝结果、修改内容、要求模型重试、降级到人工确认，或向用户说明当前无法完成。真正的产品差异往往不在于是否检测，而在于失败后能否让任务继续保持可理解和可控制。

## Use Case｜适用场景 📌

Guardrails 更适合输出会影响后续系统，或错误成本较高的 AI 功能。

在结构化信息提取中，产品可以检查字段类型、必填项和取值范围，避免错误结果直接写入数据库。

在内容生成场景中，可以围绕敏感信息、品牌规则或特定内容边界增加检查。但产品仍需评估误判，不能把任何 Validator 当成绝对正确的审核者。

在 RAG 产品中，可以检查回答是否引用上下文或是否包含必要来源。它不能证明所有事实都正确，但可以降低无来源回答直接交付的概率。

在 Agent 工作流中，模型输出可能成为工具参数。此时，格式和权限检查应发生在执行之前，避免错误内容直接转化为外部行动。

对于低风险、开放式创作任务，过多 Guardrail 可能让体验僵硬并增加延迟。规则数量应由任务风险决定，而不是越多越安全。

## Product Insight｜产品观察 🔍

Guardrails AI 最值得观察的地方，是它把 AI 产品的“失败定义”从 Prompt 中分离出来。

Prompt 描述希望模型如何回答，Guardrail 则定义哪些结果不能被产品接受。前者负责引导，后者负责检查，两者承担不同责任。

它与 Structured Outputs 也不完全相同。Structured Outputs 重点保证数据符合 Schema，适合解决字段和类型问题；Guardrails 可以继续检查结构之外的内容风险和业务规则。格式正确的答案仍然可能事实错误、包含敏感信息或不符合场景要求。

Prompt Injection 文章关注外部内容如何混淆指令与数据边界。运行时 Guardrail 可以成为其中一层防护，但不能替代权限隔离、工具确认和可信数据设计。

Agent Evaluation 则发生在开发和迭代过程中，用测试集判断系统整体表现；Guardrail 在每次真实请求中执行。离线评估帮助团队发现长期问题，运行时检查负责处理当前请求，两者不能互相替代。

另一个关键问题是，Guardrail 自身也具有不确定性。

如果 Validator 依赖模型判断，它同样可能出现误报和漏报。成熟产品需要记录被拦截内容、定期抽样复核，并允许用户或运营人员对错误判断进行反馈。

## Reusable Value｜可复用价值 🔁

设计 Guardrail 时，可以先按照风险类型选择检查方式。

格式、长度、数值和必填字段优先使用确定性规则；需要语义判断的问题再考虑分类器或模型评审；涉及重大决策和高风险操作时，保留人工确认。

每条规则还需要明确失败动作。检测到问题后是直接拒绝、自动重试、隐藏部分内容，还是交给用户决定，应该由具体场景定义。

团队可以持续观察误报率、漏报样本、拦截后的任务完成率、额外延迟和验证成本。只统计拦截次数无法判断 Guardrail 是否真正改善了体验。

用户反馈也需要被设计。清楚说明“哪项要求没有满足”和“下一步可以怎么做”，比笼统提示“内容不安全”更有助于恢复任务。

## Productization Direction｜产品化方向 🚀

### Guardrail Policy Center｜规则策略中心

团队可以按产品、用户角色和风险等级配置不同验证规则，并查看每条规则的适用范围、版本和负责人。

### Validation Trace｜验证轨迹

运行记录可以展示输入经过哪些检查、哪条规则失败、采取了什么处理，以及用户是否继续完成任务。

### Human Review Queue｜人工复核队列

对于高风险或模型无法确定的内容，产品可以进入人工复核，而不是在自动放行和直接拒绝之间二选一。

### Guardrail Evaluation｜规则效果评估

平台可以使用真实失败样本测试 Validator，比较不同规则的准确性、延迟和成本，并在上线前发现过度拦截。

Guardrails AI 提醒我们，AI 产品可靠性不是靠一条更长的 System Prompt 建立的。

真正可用的运行时控制，需要明确什么结果不可接受、如何检测，以及检测失败后怎样让用户安全地继续完成任务。
