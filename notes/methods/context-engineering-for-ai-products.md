# Context Engineering｜从 Prompt 到上下文工程 🧠

- Source｜来源：LangChain
- Type｜类型：AI 产品方法观察 / Context Engineering / Agent / RAG / Workflow
- Link｜链接：https://www.langchain.com/blog/context-engineering-for-agents

Context Engineering，通常可以翻译为“上下文工程”。

它不是简单替代 Prompt Engineering，而是把 AI 产品中的 Prompt、Memory、RAG、Tools、MCP、Workflow 和 Agent 状态管理放到同一个视角下理解。

如果说 Prompt Engineering 更关注“如何写好一条指令”，那么 Context Engineering 更关注：

> 在具体任务中，模型应该看到什么信息、什么时候看到、以什么结构看到，以及哪些信息不应该进入上下文。

从产品视角看，Context Engineering 代表了 AI 产品设计中的一个重要变化：

> AI 产品的核心不只是生成能力，而是信息环境的设计能力。

---

## What It Is｜它是什么

Context Engineering 是围绕 LLM / Agent 系统进行上下文设计、选择、压缩、隔离和管理的方法。

在传统 Prompt 使用中，用户通常关注的是：

- Prompt 怎么写
- 指令是否清楚
- 输出格式是否明确
- 模型是否理解任务

但在更复杂的 AI 产品中，Prompt 只是上下文的一部分。

一个真实 AI 产品里，模型能看到的信息可能来自：

- 用户输入
- 系统提示词
- 历史对话
- 长期记忆
- RAG 检索结果
- 工具调用返回
- MCP Server 提供的外部上下文
- Workflow 中上一步的中间结果
- 当前任务状态
- 用户画像和权限信息

这些信息共同构成了模型执行任务时的“上下文环境”。

因此，Context Engineering 关注的不是单条 Prompt，而是整个 AI 系统如何组织信息、控制信息流和管理任务状态。

---

## Why It Matters｜为什么值得关注 💡

Context Engineering 值得关注，是因为 AI 产品的失败很多时候不是模型能力不够，而是上下文设计不合理。

一个模型可能本身能力很强，但如果它拿到的是错误、混乱、过长或互相冲突的信息，最终结果仍然会不稳定。

常见问题包括：

- 模型看不到关键资料
- RAG 检索结果不相关
- 历史对话干扰当前任务
- 工具返回信息太长，淹没重点
- 多个任务状态混在一起
- Agent 多步执行后偏离原始目标
- 用户以为 AI 记得，但系统其实没有保留有效上下文

这些问题表面上看像“AI 不聪明”，但本质上很多是上下文管理问题。

LangChain 在关于 Agent Context Engineering 的文章中，把常见策略归纳为 write、select、compress、isolate 四类，也就是写入上下文、选择上下文、压缩上下文和隔离上下文。

这说明上下文已经不只是工程细节，而是 Agent 产品能否可靠运行的关键设计问题。

---

## How It Works｜它如何工作 ⚙️

Context Engineering 可以从四个动作理解。

### Write Context｜写入上下文

写入上下文，是指把任务过程中的重要信息保存下来，供后续步骤使用。

这类信息不一定都放在模型当前上下文窗口里，也可以存在外部系统中，例如：

- Memory
- Scratchpad
- 用户偏好
- 任务进度
- 中间结果
- 工具调用记录
- 项目知识库

从产品角度看，Write Context 解决的是：

> 哪些信息值得被系统记住？

这和普通聊天产品里的“记忆”不完全一样。

在 AI 产品中，记忆不只是用户说过什么，还包括任务状态、业务目标、执行过程和历史决策。

例如，一个 AI 项目助手不只要记住用户叫谁，还要记住：

- 当前项目目标
- 已完成哪些步骤
- 哪些约束不能违反
- 哪些文件已经生成
- 哪些决策已经确认

这些都是产品级上下文。

### Select Context｜选择上下文

选择上下文，是指从大量信息中挑选当前任务真正需要的部分。

这在 RAG 和 Agent 产品中很常见。

比如用户问一个问题时，系统不应该把整个知识库都塞给模型，而应该选择最相关的资料片段、工具结果或历史记录。

Select Context 解决的是：

> 当前这一步，模型真正需要看什么？

它直接影响 AI 产品的准确性和成本。

如果选择太少，模型缺信息；如果选择太多，模型容易被噪声干扰。

从产品视角看，这一步对应的是信息推荐、权限判断和任务理解。

### Compress Context｜压缩上下文

压缩上下文，是指当信息太多时，对历史内容或中间结果进行摘要、合并或结构化压缩。

这类能力在长对话、多轮任务和复杂 Agent 流程中尤其重要。

常见方式包括：

- 对历史对话做摘要
- 把长文档压缩成要点
- 把工具返回结果转成结构化字段
- 删除无关中间过程
- 保留关键决策和约束

Compress Context 解决的是：

> 如何在有限上下文窗口里保留最重要的信息？

它不是简单“缩短文本”，而是判断哪些信息对任务仍然有价值。

对 AI 产品来说，压缩质量会直接影响长任务体验。

如果压缩丢掉了关键约束，后续结果就会偏离；如果压缩保留太多无关细节，模型仍然会被干扰。

### Isolate Context｜隔离上下文

隔离上下文，是指把不同任务、不同 Agent 或不同阶段的上下文分开管理，避免互相污染。

在多 Agent 或复杂 Workflow 中，这一点很重要。

例如：

- 一个 Agent 负责检索资料
- 一个 Agent 负责写作
- 一个 Agent 负责检查事实
- 一个 Agent 负责生成执行指令

它们不一定都需要看到完整上下文。

有些 Agent 只需要看到当前任务输入，有些只需要看到结构化结果，有些只需要看到错误信息。

Isolate Context 解决的是：

> 哪些信息应该被隔离，避免影响其他任务？

从产品视角看，这对应的是任务边界、角色边界和信息权限边界。

## Product Insight｜产品观察 🔍

Context Engineering 对 AI 产品实践的启发，不只是“上下文窗口要管理好”，而是要重新理解 AI 产品的结构。

很多 AI 产品表面上是一个聊天框，但真正决定体验的是聊天框背后的信息组织方式。

用户看到的是：

> 我输入一句话，AI 给我一个回答。

但系统背后实际发生的是：

- 判断用户意图
- 选择相关知识
- 拼接系统指令
- 读取历史状态
- 调用工具或 MCP Server
- 接收工具返回结果
- 压缩中间信息
- 生成最终回答
- 决定是否写入记忆

这些环节共同决定了 AI 产品是否可靠。

因此，AI 产品设计不能只看页面流，还需要看上下文流。

---

### 1. Prompt 只是入口，不是完整系统

早期使用 AI 时，很多问题可以通过优化 Prompt 解决。

但当 AI 产品进入真实任务场景后，单条 Prompt 很快会不够用。

因为产品需要处理：

- 多轮对话
- 长期任务
- 多数据源
- 用户偏好
- 权限差异
- 工具调用
- 异常处理
- 任务恢复

这些问题无法只靠“写一句更好的 Prompt”解决。

Prompt 更像入口，Context 才是任务环境。

---

### 2. 上下文是一种产品资源

在 AI 产品中，上下文不是越多越好。

它是一种有限资源。

上下文会影响：

- 输出准确性
- 响应速度
- Token 成本
- 用户信任
- 任务连续性
- 系统可控性

因此，产品需要判断：

- 哪些信息应该被长期记住
- 哪些信息只在当前任务有效
- 哪些信息应该被压缩
- 哪些信息应该被丢弃
- 哪些信息应该被隔离
- 哪些信息需要用户确认后才能使用

这些判断本质上是产品设计问题，而不只是技术实现问题。

---

### 3. Agent 更依赖上下文管理

Agent 产品比普通 Chatbot 更依赖 Context Engineering。

原因是 Agent 通常不是一次性回答问题，而是执行多步骤任务。

它需要知道：

- 当前目标是什么
- 已经完成了哪些步骤
- 下一步应该做什么
- 可以调用哪些工具
- 工具调用结果是否可信
- 失败时如何恢复
- 什么时候需要询问用户

如果上下文管理不好，Agent 很容易出现：

- 重复执行
- 忘记目标
- 错误调用工具
- 忽略用户限制
- 把旧任务信息带入新任务
- 中途偏离原始需求

所以 Agent 产品的关键不只是“能调用工具”，而是“能在合适上下文中调用工具”。

## Reusable Value｜可复用价值 🔁

Context Engineering 可以为 AI 产品学习和实践提供几个可复用视角。

### 1. 用上下文流拆解 AI 产品

以后分析一个 AI 产品时，不只看它有哪些功能，也可以看它如何管理上下文。

可以问：

- 用户输入会进入哪里？
- 系统会读取哪些历史信息？
- 是否有长期记忆？
- 是否接入知识库？
- 是否调用外部工具？
- 工具返回内容如何进入下一步？
- 旧信息什么时候被清理？
- 不同任务之间是否隔离？

这种拆解方式比单纯看功能列表更接近 AI 产品的真实运行逻辑。

---

### 2. 用上下文质量判断 AI 体验

AI 产品体验不稳定时，可以从上下文质量角度排查。

例如：

- 回答不准：可能是检索上下文不相关
- 回答跑偏：可能是历史上下文干扰
- 长任务失败：可能是关键约束被压缩丢失
- Agent 调错工具：可能是工具上下文描述不清
- 多轮对话混乱：可能是任务状态没有隔离
- 用户不信任：可能是上下文来源不可见

这比简单说“模型不好”更具体。

---

### 3. 用上下文设计连接 RAG、MCP 和 Workflow

Context Engineering 能把几个常见 AI 产品主题连接起来：

- RAG 解决知识上下文
- Memory 解决历史上下文
- MCP 解决外部工具和数据上下文
- Workflow 解决上下文流转
- Agent 解决任务执行中的上下文决策

从这个角度看，RAG、MCP、Workflow 和 Agent 不是孤立概念，而是都在回答同一个问题：

> 如何让模型在正确的时间看到正确的信息，并完成正确的动作？

---

## Practical Use｜实践用法 🧩

如果要在 AI 产品实践中使用 Context Engineering，可以从几个简单问题开始。

### 1. 先定义任务上下文

在设计一个 AI 功能前，先明确：

- 用户要完成什么任务？
- 任务需要哪些信息？
- 哪些信息由用户输入？
- 哪些信息由系统补充？
- 哪些信息来自外部工具？
- 哪些信息来自知识库？
- 哪些信息需要被记住？

这一步可以避免直接开始写 Prompt。

### 2. 再设计上下文来源

明确任务后，可以拆分上下文来源：

- System Prompt：定义角色和边界
- User Input：表达当前需求
- Memory：保留长期偏好或历史记录
- RAG：提供知识材料
- Tools：提供实时结果或外部操作
- Workflow State：记录任务进度
- Output Schema：约束输出结构

这样可以让 AI 功能从“一个 Prompt”变成“一个可管理的信息系统”。

### 3. 最后设计上下文生命周期

上下文不是一次性塞进去就结束，还需要生命周期管理。

需要判断：

- 哪些信息只在当前轮有效
- 哪些信息要保留到任务结束
- 哪些信息要进入长期记忆
- 哪些信息需要压缩
- 哪些信息需要删除
- 哪些信息不能跨任务共享

这一步决定 AI 产品是否能长期稳定使用。

## Possible Productization｜可能的产品化方向 🚀

Context Engineering 也可以进一步产品化。

### 1. Context Console｜上下文控制台

AI 产品平台可以提供一个上下文控制台，让用户或开发者看到：

- 当前模型看到了什么
- 哪些资料来自 RAG
- 哪些信息来自 Memory
- 哪些工具结果进入了上下文
- 哪些历史内容被压缩
- 哪些上下文被丢弃

这会提升 AI 产品的可解释性和可信度。

### 2. Memory Management｜记忆管理

很多 AI 产品已经开始强调记忆，但记忆需要产品化管理。

用户应该能够：

- 查看 AI 记住了什么
- 修改错误记忆
- 删除不想保留的信息
- 控制哪些信息可跨任务使用
- 区分长期记忆和临时任务上下文

这会直接影响用户对 AI 产品的控制感。

### 3. Context Template｜上下文模板

不同 AI 场景可以沉淀不同上下文模板。

例如：

- 研究助手上下文模板
- 会议总结上下文模板
- 客服问答上下文模板
- 代码审查上下文模板
- 文档问答上下文模板
- 产品需求分析上下文模板

模板化的价值在于把上下文设计经验变成可复用资产。

### 4. Agent Context Debugger｜Agent 上下文调试器

Agent 产品需要专门的上下文调试工具。

它可以帮助观察：

- Agent 每一步看到的信息
- 每次工具调用的输入输出
- 任务状态如何变化
- 哪些信息影响了最终决策
- 错误发生在哪个上下文阶段

这类能力会成为 Agent 产品从 Demo 走向可用系统的重要基础设施。

## Product Observation｜进一步观察

Context Engineering 说明了一个趋势：

> AI 产品的关键设计对象，正在从“界面”扩展到“信息环境”。

传统产品设计更关注页面、路径、按钮、反馈和状态。

AI 产品仍然需要这些，但还需要额外关注：

- 模型看到什么
- 模型记住什么
- 模型忽略什么
- 模型调用什么
- 模型如何处理长任务
- 用户如何理解这些过程

未来 AI 产品的体验差异，可能不只来自模型本身，而是来自上下文组织能力。

一个产品如果能稳定地选择、压缩、隔离和复用上下文，就更可能把 AI 能力变成可靠的产品体验。

这也是 Context Engineering 对 AI 产品实践最重要的启发。
