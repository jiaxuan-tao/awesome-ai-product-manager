# Structured Outputs｜把模型回答变成可验证的产品数据 🧱

- Source｜来源：Google AI for Developers Docs
- Type｜类型：AI 产品方法观察 / Structured Outputs / JSON Schema / Output Validation
- Link｜链接：https://ai.google.dev/gemini-api/docs/structured-output

很多 AI 产品在原型阶段都从自然语言回答开始。

用户输入一个问题，模型返回一段文本，只要内容大致符合预期，功能就可以继续演示。但当模型输出需要进入界面组件、写入数据库、触发后续流程，或者交给另一个 Agent 使用时，“看起来像正确答案”已经不够。

产品真正需要的是一种更稳定的结果：字段是否完整、类型是否正确、状态是否明确，以及下游系统能否在不猜测文本含义的情况下继续工作。

Structured Outputs 关注的正是这一层。它让产品通过预先定义的数据结构约束模型输出，使模型回答从一段需要人工理解的文本，逐渐变成可以被程序验证和使用的产品数据。

从产品视角看，这不只是输出格式的变化，而是 AI 能力从“生成内容”进入“参与业务流程”的重要条件。

---

## What It Is｜它是什么 🧩

Structured Outputs 可以理解为给模型输出建立一个明确的数据契约。

产品不再只告诉模型“请用 JSON 返回”，而是进一步定义结果中应该有哪些字段、每个字段是什么类型、哪些字段必须存在，以及某些值只能从哪些选项中选择。

例如，一个用户反馈分析功能，最终结果可能需要包含情绪类型、问题分类、摘要和建议动作。如果只依赖自然语言 Prompt，模型可能更改字段名称、遗漏内容，或者把本应是数组的数据写成一段文字。

当产品提供 Schema 后，模型输出就被限制在更明确的结构中。Google 的 Structured Outputs 文档将数据提取、结构化分类和 Agent 工作流列为典型场景，并支持通过 JSON Schema 描述预期结果。

这并不意味着输出内容一定正确。Schema 能提高结构的可预测性，却不能自动保证事实准确、业务判断合理或字段内容满足真实需求。因此，Structured Outputs 更适合被理解为“结构可靠性的基础”，而不是“结果正确性的保证”。

---

## Why It Matters｜为什么值得关注 💡

自然语言对人很友好，但对产品流程并不总是友好。

当用户只是阅读一段总结时，表达略有变化通常不会造成严重问题。但如果模型输出要控制页面状态、生成表单、更新任务、选择工具参数或触发业务操作，格式变化就可能直接导致流程中断。

很多 AI 产品早期会用字符串截取、正则表达式或反复重试来处理这些问题。模型偶尔多说一句解释，程序就无法解析；字段名称发生变化，前端组件就显示为空；输出缺少关键值，后续工具却仍然被调用。

这些问题表面上像模型不稳定，本质上也暴露了产品没有为 AI 输出建立清楚边界。

Structured Outputs 的意义，在于把“希望模型怎么回答”进一步转化为“产品允许接收什么结果”。前者仍然主要依靠语言表达，后者开始形成可以验证的系统约束。

这让 AI 产品团队可以更准确地区分两类问题：一种是结构是否合格，另一种是内容是否正确。只有先把这两层分开，异常处理、质量评估和用户反馈才更容易被设计。

## How It Changes the Product｜它如何改变产品流程 ⚙️

Structured Outputs 真正产生价值，不是因为返回了整齐的 JSON，而是因为它改变了模型结果进入产品的方式。

### Output Contract｜建立输出契约

产品首先需要定义完成任务所必需的信息，而不是把所有判断都留给模型自由表达。

这要求团队回答：哪些字段会被用户看到，哪些字段会控制后续流程，哪些值允许为空，哪些状态必须由系统明确区分。

一个好的 Schema 实际上也是产品结构的映射。字段设计过于宽泛，输出仍然难以使用；字段设计过细，又可能让模型在不确定信息上被迫给出看似精确的结果。

因此，Schema 设计不是单纯的开发工作，也需要产品判断任务边界和信息优先级。

### Validation Layer｜增加验证层

结构符合要求之后，产品仍然需要验证内容。

日期字段可以符合字符串格式，但日期本身可能不存在；分类值可以落在枚举范围内，但模型仍可能选错类别；订单金额可以是数字，却不一定与真实数据一致。

成熟的流程应把 Schema 校验、业务规则校验和事实核对分开。结构错误可以触发重新生成或降级处理，业务冲突可以要求用户确认，高风险事实则应回到可信数据源验证。

这种分层可以避免产品把所有失败都笼统归因于“AI 回答错了”。

### Error State｜设计失败状态

结构化输出仍然可能失败，尤其是在 Schema 过于复杂、输入信息不足或任务本身存在歧义时。

产品需要提前决定失败后发生什么：自动重试、减少字段、退回文本回答、要求用户补充信息，还是暂停后续动作。

对用户而言，最重要的不是系统永远不出错，而是出错时不会生成一个看似完整、实际上不可用的结果继续向后传递。

因此，Structured Outputs 也要求产品为“不完整”“无法判断”“需要确认”等状态保留正式位置，而不是逼模型在每个字段中都给出答案。

## Use Case｜适用场景 📌

Structured Outputs 最适合结果需要被系统继续消费的场景。

在用户反馈分析中，模型可以把自然语言反馈转化为问题类型、严重程度、摘要和建议负责人，产品再将这些字段写入工单系统。这里的价值不是让总结更漂亮，而是减少人工重新分类和录入。

在文档与数据提取场景中，系统可以从合同、简历、报告或表单中提取固定字段。用户可以直接检查结构化结果，并针对某个字段修改，而不必在长文本中寻找信息。

在 Agent 工作流中，前一个步骤的输出往往会成为下一个工具或 Agent 的输入。稳定的数据结构能减少不同步骤之间对自然语言含义的重复猜测，也让任务失败更容易定位。

在 Generative UI 中，结构化结果可以映射到卡片、表格、图表或表单。但 Structured Outputs 与 Generative UI 并不相同：前者解决数据是否符合契约，后者解决这些数据如何被呈现和继续操作。

开放讨论、创意写作和情绪表达则未必需要严格结构。为了结构而结构，反而可能压缩模型在语言表达上的价值。是否使用 Structured Outputs，应由下游任务决定，而不是把它当成所有 AI 回答的默认形式。

## Product Insight｜产品观察 🔍

Structured Outputs 带来的关键产品变化，是把 AI 输出从“界面末端”变成“系统中间层”。

在普通聊天产品中，回答通常到用户阅读为止。用户负责理解、判断和决定下一步，系统不需要精确知道回答内部包含什么。

但在任务型 AI 产品中，输出可能要进入下一张页面、下一次工具调用或下一段业务流程。此时，模型回答就不再只是内容，而是一种系统状态。

这也意味着产品经理需要开始关心输出字段，而不仅是 Prompt 文案。一个字段是否允许为空、一个分类是否需要“无法判断”、一个动作是否必须经过确认，都会影响真实体验。

Structured Outputs 还揭示了模型灵活性与产品确定性之间的关系。模型擅长理解模糊输入和生成多样内容，软件系统擅长处理明确状态和稳定接口。好的 AI 产品不是让其中一方完全替代另一方，而是在合适的位置建立转换层。

用户可以用自然语言表达目标，模型负责理解语义；进入业务流程前，结果被转换为清楚的数据结构；涉及风险和歧义时，再通过界面让用户确认。

这条路径比“用户说一句话，系统直接执行”更稳健，也更接近可以长期使用的 AI 产品。

## Reusable Value｜可复用价值 🔁

在设计需要结构化输出的 AI 功能时，可以先建立一份轻量检查框架。

第一步不是立即写 Schema，而是确认下游真正需要什么。哪些字段只是展示信息，哪些字段会触发动作，哪些信息已经可以从数据库获得，不应该让模型重新猜测。

第二步是为不确定性留出表达空间。可选字段、未知状态、置信提示和人工确认都比强迫模型生成完整答案更可靠。

第三步是把验证拆成不同层次：格式是否符合要求、字段之间是否符合业务规则、内容是否有可信依据。这样，产品可以针对不同问题采用不同处理方式。

第四步是观察真实使用结果。结构成功率、重试次数、字段修改率、人工接管率和下游任务完成率，都比“JSON 是否成功返回”更能说明产品价值。

这套方法可以用于信息提取、内容审核、知识库、运营自动化、Agent 工作台和 Vibe Coding 项目。它帮助团队把模型能力逐步接入产品，而不是让整个流程依赖一段不可预测的文本。

## Product Observation｜进一步观察 🧭

AI 产品的发展正在形成一条清晰路径：Prompt 负责表达任务，Context 负责提供信息，Tools 负责执行能力，而 Structured Outputs 负责把模型结果交还给产品系统。

仓库中已有的 Prompt Versioning 关注指令如何被管理，Generative UI 关注结果如何进入动态界面，Tool Calling UX 关注执行过程如何被理解。Structured Outputs 补充的是这些环节之间的数据契约。

它看起来是一项偏工程的能力，但背后涉及的是产品如何定义结果、管理不确定性和阻止错误继续扩散。

真正成熟的 AI 产品，不会只追求模型回答更像人，也会让关键结果更容易被系统验证、被用户检查，并在必要时安全地停下来。

当模型输出能够稳定进入界面、数据和工作流时，AI 才从一个生成内容的入口，变成产品中可以被组织和管理的能力层。
