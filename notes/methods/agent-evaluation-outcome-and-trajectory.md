# Agent Evaluation｜从最终答案到执行轨迹 🧪

- Source｜来源：Anthropic Engineering
- Type｜类型：AI 产品方法观察 / Agent Evaluation / Trajectory / Outcome / Evals
- Link｜链接：https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents

普通 LLM 功能通常可以从一次输入和一次输出判断效果。

但 Agent 会在多个步骤中调用工具、修改环境、处理反馈并调整计划。最终回复看起来合理，不代表真实任务已经完成；过程出现一次错误，也可能在后续步骤中持续放大。

Agent Evaluation 关注的是如何对这种多步骤、非确定性的任务系统进行评估。

从产品视角看，它最重要的价值不是建立更多分数，而是迫使团队回答一个基本问题：

> 用户把目标交给 Agent 后，什么结果才算真正完成，哪些过程风险不能被最终一句“已完成”掩盖。

---

## What It Is｜它是什么 🧩

Agent Evaluation 是围绕任务、运行过程和最终状态建立的测试方法。

Anthropic 在官方文章中将一次评估拆成 Task、Trial、Grader、Transcript、Outcome 和 Evaluation Harness 等部分。

Task 定义需要完成的目标和成功标准；Trial 是 Agent 对任务的一次实际尝试；Transcript 或 Trajectory 记录消息、工具调用和中间结果；Outcome 则是执行结束后的真实环境状态。

例如，一个预订 Agent 最后回复“机票已经预订”，这只是 Transcript 的一部分。真正的 Outcome 是预订系统中是否存在正确订单，日期、乘客和价格是否符合用户要求。

这种区分让评估从“回答像不像正确答案”转向“任务系统是否产生了正确结果”。

---

## Why It Matters｜为什么值得关注 💡

Agent 的能力越强，评估难度越高。

它可能选择不同工具，以不同顺序完成同一个目标。只检查固定步骤，会把合理的新路径判定为失败；只检查最终文字，又可能忽略越权调用、重复操作和隐藏错误。

模型输出还具有非确定性。同一个任务运行多次，可能成功一次、失败一次。单次演示很难说明产品是否稳定，更无法判断一次 Prompt、模型或工具修改是否造成回退。

没有系统评估时，团队容易进入被动循环：用户报告 Agent 变差，开发者手动复现，修改某项配置，再依靠感觉判断是否改善。

Agent Evaluation 值得关注，是因为它把产品要求转化成可重复运行的任务与判断标准。

早期建立 Eval 可以帮助团队明确产品边界；进入生产后，它又可以用来比较模型、检测回归，并判断新能力是否值得上线。

## Evaluation Structure｜评估结构 ⚙️

从产品实践看，Agent Eval 可以围绕任务设计、过程记录和结果评分三个层次展开。

### Task and Success Criteria｜任务与成功标准

好的测试任务应来自真实用户目标，并明确完成条件。

“帮用户处理退款”仍然过于模糊。产品需要定义是否确认了订单、退款金额是否正确、是否遵守政策，以及用户最终是否收到结果。

如果成功标准无法写清，通常说明产品需求本身还不够明确。

### Trajectory and Outcome｜轨迹与结果

Trajectory 帮助团队理解 Agent 如何完成任务，包括使用了哪些工具、经过多少轮、在哪里遇到错误以及是否请求用户确认。

Outcome 则验证任务结束后真实世界发生了什么。

评估不应强迫 Agent 逐步复制唯一流程，但需要检查关键边界。例如，结果正确却调用了未经授权的工具，仍然不能被视为完全成功。

### Graders｜评分方式

官方文章将常见 Grader 分成代码规则、模型评审和人工评审。

确定性结果应优先使用代码或环境状态验证，例如测试是否通过、记录是否创建。开放式质量可以使用带清楚 Rubric 的模型评审，但需要与人工判断校准。涉及主观体验或专业风险时，仍需要人类评审。

多种 Grader 组合通常比依赖单一总分更能解释问题。

## Use Case｜适用场景 📌

不同 Agent 需要不同成功标准。

Coding Agent 可以通过测试、静态检查和文件差异验证结果，同时观察是否修改了无关代码、是否重复尝试或绕过约束。

研究型 Agent 更难依赖单一标准答案，可以检查来源质量、事实覆盖、引用一致性和结论是否由资料支持。

客服或业务 Agent 既要验证工单、退款或预约是否真实完成，也要评估沟通过程是否清楚、是否在必要节点获得用户确认。

Computer Use Agent 则需要在隔离环境中执行任务，检查最终页面或系统状态，并记录中途是否点击了错误对象或暴露敏感信息。

如果功能只是一次低风险文本生成，完整 Agent Eval Harness 可能过重。评估复杂度应与任务步骤和失败成本匹配。

## Product Insight｜产品观察 🔍

Agent Evaluation 最重要的产品启发，是评价对象已经从“模型回答”变成了“模型、工具、环境和编排共同组成的系统”。

一次失败可能来自模型理解错误、工具返回异常、权限不足、环境状态残留或工作流设计问题。只更换模型并不一定解决任务失败。

它与现有的 AI Product Evaluation 文章处于不同层次。AI Product Evaluation 提供用户场景、任务完成、信任和异常兜底等通用观察维度；Agent Evaluation 进一步把这些维度转化为可重复执行的测试任务和评分逻辑。

它也不同于 Langfuse。Langfuse 是可观测性与评估平台案例，帮助记录 Trace、Prompt 和指标；本篇关注的是团队应该定义什么任务、读取哪些运行记录，以及如何判断成功。工具能够承载方法，但不能替团队决定产品标准。

另一个关键观察是，不稳定性本身需要被测量。

Agent 在一次测试中成功，并不代表用户每次都能获得相同结果。团队需要运行多次 Trial，观察成功概率和一致性，而不是只保留最好的一次演示。

## Reusable Value｜可复用价值 🔁

建立 Agent Eval 时，可以先从少量真实任务开始，而不是追求庞大 Benchmark。

每个任务记录用户目标、初始环境、允许工具、关键限制和最终成功状态。然后选择最便宜且最可靠的 Grader：能用环境状态验证的，不用模型猜测；需要语义判断的，建立清楚 Rubric 并保留人工抽样。

评估集还应区分能力测试和回归测试。

能力测试用于探索 Agent 目前还做不好的任务，为产品寻找改进方向；回归测试用于保护已经稳定的能力，避免模型、Prompt 或工具更新破坏原有体验。

失败后需要阅读 Trajectory，而不是只看总分。执行记录能够帮助团队判断是 Agent 失败，还是任务描述和 Grader 本身存在问题。

Eval 也需要持续维护。真实用户出现的新失败应转化成测试案例，已经失去区分度的任务则需要调整。

## Productization Direction｜产品化方向 🚀

### Scenario Library｜任务场景库

产品、运营和领域专家可以共同维护真实任务、边界条件和成功标准，让评估不只由工程团队定义。

### Trajectory Review｜执行轨迹复盘

评估平台可以并排展示工具调用、状态变化、Grader 结果和失败原因，帮助团队定位问题发生在哪一步。

### Release Gate｜发布质量门槛

模型、Prompt、工具或工作流更新前，系统可以运行回归任务，并根据关键场景表现决定是否发布。

### User Feedback to Eval｜反馈转化测试

用户投诉和人工复盘可以被整理成新的 Eval Task，使真实失败逐渐成为产品质量资产。

Agent Evaluation 说明，Agent 产品不能只依赖成功演示证明价值。

当任务成功被明确、执行过程可以复盘、结果能够重复验证，团队才有基础判断 Agent 是否真的变得更可靠。
