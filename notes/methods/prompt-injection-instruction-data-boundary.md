# Prompt Injection｜AI 产品中的指令与数据边界 🛡️

- Source｜来源：OWASP GenAI Security Project Official Website
- Type｜类型：AI 产品方法观察 / Prompt Injection / Agent Safety / RAG Security
- Link｜链接：https://genai.owasp.org/llmrisk/llm01-prompt-injection/

当 AI 产品只回答一段文本时，Prompt Injection 可能表现为跑题、泄露内部提示或生成不当内容；当产品接入 RAG、网页、文件、邮件、MCP 与业务工具后，同一种问题还可能继续转化为数据外泄、错误决策和未授权操作。

它的核心并不是“用户写了一条坏 Prompt”，而是模型通常在同一自然语言上下文中同时处理任务指令与待分析数据。网页正文、文档段落和工具返回值原本应当只是材料，却可能被模型解释为新的命令。于是，产品真正要设计的不是一条更强硬的系统提示词，而是指令可以来自哪里、数据允许影响什么，以及模型受到影响后仍然能够做什么。

OWASP 将 Prompt Injection 列为 LLM 应用的主要风险之一，同时明确指出：RAG 与微调可以改善相关性和准确性，但不能彻底消除注入风险。由于生成式模型本身具有概率性，目前也没有被证明能够一劳永逸阻止所有 Prompt Injection 的单一方案。更现实的产品目标，是通过分层防护降低成功概率，并把可能造成的影响限制在可发现、可中止和可恢复的范围内。

---

## What It Is｜它是什么 🧩

Prompt Injection 是指输入改变模型行为或输出，使其偏离产品原本设定的目标、约束或权限边界。输入不一定是人类可见的明确命令，也可能经过编码、拆分，隐藏在图片、网页样式、文档元数据或多轮上下文中；只要最终被模型解析并影响行为，就可能构成注入。

它与 Jailbreaking 有关联，但不完全相同。Jailbreaking 通常特指诱导模型绕过安全策略，是 Prompt Injection 的一种表现；AI 产品还要面对另一类更广泛的问题：攻击者不必让模型“解除所有限制”，只需让它错误总结一份材料、选择错误工具、泄露一段上下文，或把不可信内容带入后续动作，就可能达到目的。

### Direct Prompt Injection｜直接注入

直接注入来自用户与模型交互的入口。用户可能明确要求模型忽略原有规则、暴露系统提示或调用不应使用的能力，也可能无意中提交一段与产品指令冲突的文本。它通常更容易被注意，却不能只靠关键词拦截：攻击可以改写、编码、跨语言表达或分散到多轮对话中，正常业务内容也可能包含“忽略以上内容”之类需要被分析而不是执行的句子。

### Indirect Prompt Injection｜间接注入

间接注入来自模型为完成任务而读取的外部内容。例如，一个研究助手打开网页，一个文档助手读取 PDF，一个邮件助手处理正文和附件，一个 RAG 系统检索知识库，一个 Agent 接收搜索、数据库或 MCP 工具的结果。攻击指令可能由内容发布者主动植入，也可能来自被污染的数据源，甚至由上游系统无意产生。

间接注入更能暴露产品边界问题，因为发起任务的用户可能完全没有输入恶意内容。用户只说“总结这个网页”，真正改变模型行为的文字却藏在网页中。此时，网页是否相关与网页是否可信是两个不同判断；检索命中率高，也不代表检索结果有权重新定义任务。

## Why It Matters｜为什么是产品问题 ⚠️

AI 产品正在不断扩大模型可见的信息和可执行的能力。RAG、浏览器、文件上传和工具调用提高了任务完成度，也让不可信数据更容易进入决策链。风险并不只存在于公开互联网：企业知识库可能包含外部复制内容，用户文件可能带有隐藏层，数据库字段可能由第三方填写，工具结果也可能携带用户可控文本。即使来源经过身份认证，其中的每一段内容也不应自动获得指令级信任。

因此，风险大小主要由业务场景与模型的 Agency 决定。一个只生成草稿的助手被影响后，损失可能是低质量内容；一个能读取客户数据、发送邮件、修改订单或运行代码的 Agent 被同样影响，后果会显著扩大。模型是否被诱导是一层问题，被诱导后能接触哪些数据、调用哪些能力、结果是否会自动生效，是产品必须控制的另一层问题。

这也解释了为什么“模型拒绝了测试攻击”不能等同于产品安全。一次拒绝只说明某个样本在某个版本下没有成功，无法证明变体、间接来源、多轮组合或未来模型版本仍然失败。安全性需要落到系统架构、执行规则和运营机制，而不能只停留在对话表现。

## Product Boundary｜与相邻主题的边界 🧭

### 与 Context Engineering 的区别

Context Engineering 关注模型在某一步应该看到什么、何时看到、如何选择、压缩和隔离，核心是上下文的相关性、质量与生命周期。Prompt Injection 关注的是其中的信息是否被赋予了错误的控制力。一个 RAG 片段可以非常相关，却仍是不可信数据；上下文组织正确，并不代表指令与数据边界安全。两者会使用相似的来源标记和隔离机制，但回答的是不同问题。

### 与 MCP Authorization 的区别

MCP Authorization 关注 AI 以谁的身份连接外部系统、被授予哪些 Scope，以及授权如何过期和撤销，核心是访问权限。Prompt Injection 关注模型为什么决定发起某次访问或操作，以及这个决定是否被不可信内容劫持。最小权限可以缩小注入成功后的影响范围，却不能证明一次工具调用符合用户原始意图；反过来，检测到可疑文本也不能代替正式授权。

### 与 Tool Calling UX 的区别

Tool Calling UX 关注如何向用户呈现动作、参数、进度、结果与失败，让执行过程可理解、可控制。Prompt Injection 提供的是上游威胁模型：哪些动作不应自动执行，哪些参数必须由确定性规则验证，哪些场景必须进入确认。确认界面本身属于 Tool Calling UX，而在受到不可信内容影响时设置不可绕过的执行门槛，属于 Prompt Injection 的风险控制。

## Defense in Depth｜从边界到分层防护 🛠️

有效防护不是堆叠一组拦截器，而是沿着攻击链设置彼此独立的限制：不可信内容进入上下文后，先限制它改变任务的机会；即使模型判断被影响，也限制它获得敏感信息和高风险能力；即使生成了危险输出，也阻止输出直接进入界面、工具和业务系统；如果仍然发生异常，则能够定位来源、停止扩散并恢复状态。

### 进入上下文前：标明来源，但不把标签当保证

系统应区分开发者控制的指令、用户目标和外部数据，把网页、文件、RAG 片段、邮件正文、工具返回与历史记忆标为不可信输入，并使用清晰的结构和分隔告诉模型这些内容只能被分析，不能改写任务。外部内容进入模型前，还可以移除活动脚本、隐藏标记和无关元数据，检查编码与异常模式，并对高风险来源增加专门分类或隔离处理。

这些措施可以降低风险，却不是强安全边界。自然语言标签仍由模型解释，规则过滤可能被改写和混淆绕过，负责检测的模型本身也可能出错。因此，“已清洗”不应被转换成“可以自动执行”，输入防护必须与后续能力控制配合。

### 模型做决定时：能力最小化比提示词更基础

应用只应向当前任务暴露必要的数据和工具，并把读取、写入、发送、删除等能力拆开。只读任务不应附带写权限；一次性任务优先使用短期、窄范围凭证；批量操作、跨空间访问和不可逆动作不应成为默认能力。API 密钥、系统凭证和无关敏感数据也不应进入模型上下文。

关键业务约束应由确定性代码执行，而不是让模型自行遵守。例如，工具白名单、用户与资源权限、参数范围、收件人域、金额上限和可修改字段，都应在调用前重新校验。这样即使模型接受了注入指令，它能提出的动作仍会被应用边界截断。

### 输出进入系统前：验证、确认，再执行

模型输出同样是不可信输入。面向用户展示前，应检查敏感信息、危险链接、可执行 Markdown 或 HTML；作为下游数据时，应同时验证结构、业务规则和事实依据；作为工具参数时，还要与用户原始目标、当前会话权限和资源范围比对。Structured Outputs 能让格式更可验证，但不能保证内容正确或意图没有偏移。

人工确认应放在模型提出动作之后、真实副作用发生之前，重点覆盖发送、发布、支付、删除、权限变更和大范围写入等高风险操作。有效确认不是泛化的“是否继续”，而是让用户看到真实目标、关键参数、影响范围、数据来源与可逆性。人工并非用来替系统逐条审查所有内容，而是作为高风险路径上独立于模型判断的最后授权者。

### 运行之后：让异常可观测、可中止、可恢复

产品需要记录一次任务使用了哪些数据来源、经过哪些模型与规则、提出和执行了哪些工具调用、何处触发拦截或人工确认，同时避免把密钥和完整敏感内容写入日志。来源标识、Prompt 与模型版本、策略版本、工具参数摘要和最终状态应能串成同一条 Trace，帮助团队判断问题来自直接输入、被污染的检索结果，还是工具返回。

观测的目的不只是告警，还要支持恢复。当可疑行为出现时，系统应能暂停当前运行、隔离被污染的文档、撤销短期凭证、关闭相关工具、清除受影响的记忆或缓存，并对可逆操作执行回滚。直接与间接注入、编码与多语言变体、隐藏内容、多轮延迟触发和多模态输入也应进入持续对抗测试与版本回归，而不是上线前只测几句典型攻击语。

## Product Insight｜产品观察 🔍

Prompt Injection 让 AI 产品的信任边界从“谁可以访问系统”进一步扩展到“哪些内容可以影响决策”。一份通过正规账号上传的文件，在身份层面可能可信，在内容层面仍可能不可信；一个由官方工具返回的结果，在传输层面可能可靠，其中的文本仍可能来自外部用户。产品需要把信任附着到数据来源、内容类型和允许影响的动作上，而不是给整个渠道贴一个永久的可信标签。

它也提示团队重新看待自动化程度。更强的模型、更长的上下文和更多工具会提高能力，但 Agency 增加的同时，攻击后的影响范围也会扩大。安全设计不是简单减少功能，而是让自动化与风险匹配：低风险、可逆、窄范围的动作可以更顺畅；涉及敏感数据、外部通信和不可逆后果时，系统必须收紧权限、加强验证并保留人的最终决定。

## Reusable Value｜可复用价值 🔁

设计这类产品时，可以把一次任务画成“指令来源 → 不可信数据 → 模型决策 → 可用能力 → 外部影响”的链路。重点不是列出所有攻击语句，而是找出数据在哪一步可能获得不应有的指令权，模型在哪一步可能接触超出任务需要的能力，以及错误结果在哪一步会从文本变成真实副作用。

成熟的 Prompt Injection 设计不会承诺模型永远不受操纵，而会保证任何单一防线失效时，后面仍有独立约束。最终目标不是得到一个“从未被攻击成功”的演示，而是建立一套能够限制影响、留下证据、及时中止并恢复业务的产品机制。

对 AI 产品而言，真正重要的边界不是界面上哪段文字被称为 Prompt、哪段被称为资料，而是谁有权改变任务、谁有权触发动作，以及这种权力如何被系统而不是模型单独约束。
