# MCP Elicitation｜让 AI 在工具调用中向用户补充信息 🗣️

- Source｜来源：Model Context Protocol Specification
- Type｜类型：AI 产品交互观察 / MCP Elicitation / Structured Input / Human-in-the-loop
- Link｜链接：https://modelcontextprotocol.io/specification/2025-11-25/client/elicitation

当 AI 通过 MCP 调用外部工具时，并不是每个任务都能依靠用户最初的一句话直接完成。

工具可能缺少必要参数，外部服务可能需要用户补充账号信息，某些流程也可能必须离开当前客户端，在可信页面中完成授权或支付。

过去，这类情况经常被简化成模型在聊天里追问一句话。但自由文本不一定能满足工具对数据格式的要求，也很难处理敏感信息、拒绝操作和任务恢复。

MCP Elicitation 提供了一种标准化机制：MCP Server 可以通过 Client 向用户请求额外信息，同时由 Client 控制具体交互界面和数据分享方式。

从产品视角看，它解决的不是“AI 会不会提问”，而是：

> 当任务缺少必要信息时，AI 产品如何在不中断上下文的前提下，让用户理解为什么需要补充、应该提供什么，以及拒绝后还能做什么。

---

## What It Is｜它是什么 🧩

Elicitation 可以理解为 MCP 工作流中的用户信息请求机制。

一个 MCP Server 在执行工具或访问外部服务时，如果发现现有上下文不足，可以向 MCP Client 发起 Elicitation 请求。Client 再选择适合自己的界面形式，把请求展示给用户，并将用户决定返回给 Server。

这里的角色边界很重要。Server 负责说明需要什么信息以及为什么需要，Client 负责实际的人机交互。协议没有要求所有产品都使用同一种弹窗或表单，因此聊天产品、桌面助手和企业工作台可以保留各自的体验方式。

MCP 官方规范目前提供 Form 和 URL 两种模式。

Form Mode 用于在 Client 内收集结构化信息。Server 可以通过受限的 JSON Schema 描述字段、类型、默认值和校验要求，Client 根据这些信息生成合适的输入界面。

URL Mode 则把用户引导到外部可信页面，适合 API Key、第三方授权或支付等不应经过 MCP Client 的敏感交互。用户在外部页面完成操作后，工作流再根据完成通知或用户操作继续推进。

---

## Why It Matters｜为什么值得关注 💡

Agent 产品经常强调“用户只需要描述目标”，但真实任务很少能在一开始就具备全部信息。

例如，用户要求预订一次会议，系统可能仍需要日期范围、参会人和会议时长；用户要求查询某个业务系统，MCP Server 可能还没有第三方授权；用户要求创建内容，工具可能需要确认输出语言、格式或目标空间。

如果系统直接猜测缺失信息，可能产生错误操作。如果把所有问题都退回普通聊天，用户又需要理解工具参数，并承担把答案整理成正确格式的工作。

Elicitation 的价值在于，它让“补充信息”成为任务工作流中的正式状态。

系统不只是告诉用户“还缺东西”，而是明确当前任务为什么暂停、需要哪些输入、数据会交给谁，以及提交后如何继续。这能减少模型反复追问，也让工具调用从一次性请求变成可交互的任务过程。

对于敏感信息，URL Mode 还建立了更清楚的数据边界。凭证和支付信息可以直接进入外部服务页面，不需要经过模型上下文或 MCP Client，从产品上减少用户对数据去向的不确定感。

## How It Works｜它如何工作 ⚙️

从产品体验看，Elicitation 可以拆成提出请求、用户决策和恢复任务三个阶段。

### Request｜提出请求

Server 发起请求时，需要提供一段用户可理解的说明。Form Mode 还会提供预期的数据结构，URL Mode 则提供外部地址和用于跟踪本次交互的标识。

这一步不应只展示技术字段。用户真正需要理解的是：哪个 Server 正在请求信息，这些信息将用于什么任务，哪些字段必须填写，以及数据是否会离开当前产品。

请求说明越清楚，用户越容易判断是否继续。反过来，如果产品只显示“工具需要更多权限”或“请输入参数”，Elicitation 即使技术上完整，也很难形成可信体验。

### Decision｜处理用户决策

MCP Elicitation 将用户结果区分为 Accept、Decline 和 Cancel。

Accept 表示用户明确提交；Decline 表示用户明确拒绝；Cancel 则表示用户关闭、退出或没有作出明确决定。这三个状态不能被产品当成同一种“失败”。

用户拒绝时，系统可以解释哪些能力因此不可用，并提供范围更小的替代方案。用户取消时，系统可以保留当前任务，让用户稍后继续，而不是立即判定整个任务失败。

这种区别看似细小，却决定了 Agent 是否尊重用户意图。成熟的工作流不应把关闭弹窗理解成默认同意，也不应在用户拒绝后不断重复请求。

### Resume｜恢复任务

提交信息不等于任务已经完成。Server 需要把新信息与原任务状态关联，继续执行尚未完成的步骤。

Form Mode 通常可以直接使用返回的结构化内容。URL Mode 则可能需要等待外部流程完成通知，也需要为通知未返回、页面加载失败或用户暂时离开提供重试和取消入口。

因此，Elicitation 不只是一个输入表单，还依赖任务状态管理。产品需要明确用户回到哪里、哪些步骤已经保留，以及是否还需要重新执行之前的操作。

## Use Case｜适用场景 📌

Elicitation 适合任务已经开始，但工具或外部服务缺少必要信息的场景。

在日历、表单和业务流程中，Form Mode 可以收集日期、选项、数量和确认范围等结构化数据。与自由文本相比，字段和校验能够降低理解歧义，也方便用户在提交前检查。

在连接第三方服务时，URL Mode 可以承接 OAuth、API Key 配置和支付流程。敏感信息直接交给对应服务，MCP Client 只负责引导和恢复任务，而不需要读取凭证内容。

在企业 Agent 中，Elicitation 还可以用于补充项目、部门或数据范围。用户不必一开始填写一张复杂配置表，系统只在执行到相关步骤时请求真正需要的信息。

但 Elicitation 不适合被滥用为频繁弹窗。如果 Server 可以从已有上下文安全获得答案，就不应重复打断用户；如果请求涉及高风险动作，单纯收集输入也不能替代明确的授权和执行确认。

## Product Insight｜产品观察 🔍

MCP Elicitation 最值得关注的地方，是它把“AI 追问”从对话技巧转变成了可设计的产品状态。

普通聊天追问主要关注语言是否自然。工具工作流中的追问还需要关注数据结构、来源身份、敏感边界和任务恢复。用户回答的不只是一个问题，而是在决定是否让外部系统获得完成任务所需的信息。

它与仓库中已有的 MCP Authorization 和 Human-in-the-loop 有明确区别。

MCP Authorization 定义 Client 与 Server 之间较长期的访问关系；Human-in-the-loop 关注高风险动作是否需要人工批准、修改或拒绝；Elicitation 则发生在任务缺少必要信息、Server 无法继续的时候。

三者共同构成 Agent 产品中的用户控制链路：先明确系统拥有什么权限，再在任务中补充必要信息，最后对关键动作保留人工决定。

另一个重要观察是，Client 不应只是机械渲染 Server 提供的字段。它需要让用户看到请求来源、数据用途和影响范围，并对可疑或过度请求保持防护。协议提供连接方式，最终信任仍然由产品体验建立。

## Reusable Value｜可复用价值 🔁

设计需要用户补充信息的 AI 工作流时，可以先判断三个问题。

第一，系统缺少的是普通任务参数，还是敏感凭证。普通参数可以在产品内结构化收集，凭证和支付信息则应优先进入外部可信页面。

第二，这次请求是否真正必要。产品应尽量复用已经获得且仍然有效的信息，避免同一个任务中反复询问，也不能为了方便而扩大收集范围。

第三，用户不提交时任务如何继续。Decline 和 Cancel 都需要清楚的下一步，例如缩小任务范围、改用其他工具、保存当前进度或稍后重试。

这套判断方式不仅适用于 MCP，也适用于任何需要 Agent 在执行中向用户补充信息的产品。

## Productization Direction｜产品化方向 🚀

### Contextual Input Panel｜上下文输入面板

产品可以根据当前任务动态展示少量必要字段，并说明它们将影响哪一步。用户提交后，界面回到原任务，而不是重新开始一次对话。

### Sensitive Action Handoff｜敏感流程转交

对于授权、支付和凭证设置，产品可以提供明确的外部转交状态：正在等待、已经完成、需要重试或用户已取消。这样，离开当前界面不会让任务失去上下文。

### Elicitation History｜信息请求记录

企业 Agent 可以记录哪个 Server 在何时请求了什么类型的信息、用户作出了什么决定，以及这些信息被用于哪个任务。记录重点应是请求和决策，不应暴露不必要的敏感内容。

MCP Elicitation 说明，Agent 的主动性不只是主动调用工具，也包括在无法安全继续时主动停下来、向用户提出清楚的问题。

真正成熟的任务系统，不会通过猜测掩盖信息缺口，而会把补充信息设计成一个可理解、可拒绝、可恢复的协作过程。
