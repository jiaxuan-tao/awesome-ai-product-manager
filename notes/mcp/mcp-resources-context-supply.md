# MCP Resources｜让 AI 应用读取可发现的上下文 📚

- Source｜来源：Model Context Protocol Specification
- Type｜类型：AI 产品方法观察 / MCP Resources / Context Supply / Application-controlled Context
- Link｜链接：https://modelcontextprotocol.io/specification/2025-11-25/server/resources

讨论 MCP 时，人们往往最先想到工具调用。

模型可以通过 MCP 调用搜索、数据库、代码仓库和业务系统，从回答问题进一步走向执行任务。但外部系统并不只提供可以执行的动作，也包含大量需要被读取和理解的信息，例如文件内容、数据库结构、项目历史、产品文档和用户配置。

MCP Resources 处理的是后一类问题：服务器如何把上下文数据以标准方式暴露给 AI 应用，由应用决定何时发现、选择和读取。

从产品视角看，Resources 的价值不在于“又增加一种协议能力”，而在于它让上下文供给成为一个可以被组织和管理的产品对象。

> Agent 不仅需要知道自己能做什么，也需要在行动前获得正确的信息环境。

---

## What It Is｜它是什么 🧩

MCP Resources 是服务器向客户端提供的上下文数据。

根据 MCP 官方规范，Resource 通过 URI 标识，可以包含文本或二进制内容。服务器可以列出固定资源，也可以通过 Resource Template 描述带参数的资源地址；客户端读取后，再决定如何把内容呈现给用户或加入模型上下文。

例如，一个代码仓库服务器可以暴露项目文件和提交历史，一个数据库服务器可以提供表结构，一个知识系统可以提供文档或知识条目。

这些内容与 Tool 有明显区别。

Tool 是模型可以请求执行的操作，例如发送消息、修改文件或查询接口；Resource 更像可被读取的上下文来源。官方概念文档将 Resources 描述为由应用控制的被动数据，而 Tools 通常由模型根据任务判断是否调用。

这种控制差异非常重要。它意味着应用可以在内容进入模型之前，决定哪些资源可见、何时读取，以及用户是否需要参与选择。

---

## Why It Matters｜为什么值得关注 💡

很多 AI 产品把上下文处理隐藏在后台。

用户上传文件、连接知识库或打开项目后，系统自动收集资料并发送给模型。用户通常不知道模型看到了哪些内容，也不知道某份资料是否已经过期、是否与当前任务相关。

当上下文来源增加，这种不透明会带来几个问题。

首先，更多信息不一定带来更好结果。无关资料会占用上下文，让真正重要的信息变得不突出。

其次，内容来源会影响信任。模型引用的是官方文档、历史记录，还是一个用户无权查看的内部文件，产品需要清楚区分。

再次，上下文也有生命周期。文件会更新，数据库结构会变化，项目状态会推进。产品不能把读取过一次的内容永久当成当前事实。

Resources 值得关注，是因为它让“上下文从哪里来”拥有明确接口。应用可以围绕资源发现、选择、读取和更新建立产品逻辑，而不是把所有外部信息都临时拼进 Prompt。

## How It Works｜它如何工作 ⚙️

从产品结构看，MCP Resources 可以拆成发现、读取和更新三个环节。

### Discovery｜发现资源

客户端可以获取服务器提供的资源列表，也可以使用 Resource Template 表达动态资源。

固定列表适合数量有限、边界清楚的内容，例如常用说明文件。模板则适合需要参数才能定位的内容，例如某个仓库中的文件、某位用户的日历或某个数据库对象。

发现机制决定了产品如何展示上下文入口。如果服务器返回大量技术 URI，普通用户很难理解；客户端需要把资源名称、来源、类型和适用范围转化成可选择的信息结构。

### Reading｜读取内容

资源被选择后，客户端通过 URI 读取具体内容。

这一阶段需要处理内容类型、大小、权限和失败状态。某个资源存在于列表中，不代表它始终可读取；文件可能被删除，账号权限可能变化，内容也可能超过合理的上下文范围。

产品需要让读取失败可见，并决定是提示用户、更换资源，还是继续完成一个信息不完整的任务。

### Updates｜处理变化

官方规范还提供资源列表变化通知与可选的订阅能力，用于让客户端知道资源或内容已经发生变化。

从产品角度看，这使 Resources 不只是静态附件。对于持续运行的 Agent，项目状态、文档和业务数据都可能在任务过程中更新，客户端需要避免继续使用已经失效的内容。

## Use Case｜适用场景 📌

MCP Resources 适合需要从外部系统持续获取上下文，但不一定立即执行操作的产品。

在代码协作场景中，应用可以读取仓库文件、项目说明和提交历史，为代码分析或任务规划提供背景。真正修改文件仍然可以通过 Tool 完成，两类能力保持清楚分工。

在数据分析场景中，数据库结构、指标定义和数据字典可以作为 Resources。模型先理解数据环境，再调用查询工具获取具体结果，能够减少在不了解结构时盲目生成查询的情况。

在知识工作场景中，用户可以从项目文档、会议记录和政策资料中选择本次任务需要的上下文，而不是让系统默认读取所有连接内容。

如果数据只需要通过一次工具查询返回，或者信息规模很小，未必需要单独设计 Resource。Resources 更适合有明确来源、可重复读取，并需要被发现或管理的上下文。

## Product Insight｜产品观察 🔍

MCP Resources 最有价值的启发，是它把上下文从“Prompt 的隐藏组成部分”变成了产品中的信息资产。

Context Engineering 关注模型为了完成任务应该看到什么；Resources 进一步提供了一种标准化供给方式。但协议只解决数据如何被暴露，并不会自动判断哪些内容真正相关。

产品仍然需要回答：由用户选择还是系统推荐？读取内容前是否展示来源？多份资料冲突时如何处理？敏感资源是否需要额外授权？内容变化后是否重新生成结果？

这些问题说明，上下文体验不能完全隐藏在 Agent 后台。

用户未必需要看到每个协议调用，但应该理解当前任务使用了哪些资料，并能够添加、移除或更换关键来源。对高风险任务来说，来源透明度甚至比模型回答的流畅度更重要。

另一个观察是，Resources、Tools 和 Prompts 代表不同控制关系。

Resources 由应用组织上下文，Tools 让模型请求行动，Prompts 则让用户调用预设工作方式。清楚区分三者，有助于产品确定用户、应用与模型分别应该控制什么，而不是把所有能力都交给 Agent 自主决定。

## Reusable Value｜可复用价值 🔁

设计外部上下文能力时，可以先建立资源清单，记录每种内容的来源、负责人、更新频率、权限、大小和可信等级。

然后再决定哪些内容适合始终加载，哪些需要用户选择，哪些只在特定任务中按需读取。这个过程能够减少“连接了数据源就全部塞给模型”的粗放做法。

资源界面也应保留基本可解释性。名称、来源、更新时间和读取状态比原始 URI 更适合用户理解；当内容过期或不可访问时，产品应明确反馈，而不是让模型在缺少资料时继续给出确定答案。

评估 Resources 体验时，可以观察资源选择是否准确、无关内容是否减少、上下文错误能否被定位，以及用户是否知道答案使用了哪些资料。

## Productization Direction｜产品化方向 🚀

### Context Library｜上下文资源库

AI 工作台可以把不同 MCP Server 提供的 Resources 组织成统一资源库，并按项目、来源、权限和更新时间展示。

用户选择的是可理解的文档与数据，而不是协议层的 URI。

### Context Picker｜任务上下文选择器

在开始任务前，产品可以推荐相关资源，并允许用户确认、移除或补充。推荐负责降低选择成本，用户控制则避免系统错误读取敏感或无关内容。

### Resource Freshness｜资源新鲜度管理

产品可以记录资源最后更新时间和读取状态，在关键内容变化后提醒用户重新执行分析或更新结论。

MCP Resources 说明，Agent 产品的基础能力不只有工具数量。

当外部信息能够被清楚发现、选择、读取和更新，模型才更有可能在正确的信息环境中完成任务。
