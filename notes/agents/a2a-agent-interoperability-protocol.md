# A2A Protocol｜从单个 Agent 到跨系统协作 🔗

- Source｜来源：A2A Protocol Specification / a2aproject/A2A
- Type｜类型：AI Agent 方法观察 / Agent Interoperability / Multi-agent Collaboration / Open Protocol
- Link｜链接：https://a2a-protocol.org/latest/specification/

很多 Agent 产品仍然运行在相对封闭的系统中。

一个 Agent 可以使用模型、调用工具、访问知识库，也可以在内部工作流中把任务交给不同角色。但当它需要与另一个团队、另一个平台或另一种框架构建的 Agent 协作时，问题会明显变复杂。

它如何知道对方能做什么？如何发起任务？如何持续获得进度？对方返回的是一条消息、一个文件，还是可以继续处理的结构化结果？如果任务需要很长时间，双方如何保持状态？

A2A，也就是 Agent2Agent Protocol，尝试为这些问题提供一种开放的交互方式。

从产品视角看，A2A 的价值不只是“让两个 Agent 能互相发消息”，而是让独立 Agent 之间可以发现能力、委派任务、交换结果，并在不暴露内部实现的情况下形成协作关系。

它标志着 Agent 产品开始从单个应用内部的能力编排，走向跨系统的服务协作。

---

## What It Is｜它是什么 🧩

A2A 是面向独立 Agent 系统的通信与互操作协议。

官方规范将它描述为一种帮助不同框架、不同语言和不同提供方构建的 Agent 进行通信的共同交互模型。协作双方不需要共享内部 Prompt、记忆、工具实现或推理过程，而是通过公开能力和任务接口完成合作。

这个边界很重要。

现实中的 Agent 不一定属于同一个平台，也不一定能被统一编排。一个研究 Agent 可能由企业内部团队维护，一个数据分析 Agent 可能来自第三方服务，一个执行 Agent 可能运行在另一套基础设施中。

如果每两个 Agent 之间都需要单独设计连接方式，系统会很快形成大量难以维护的定制接口。A2A 希望把发现、消息、任务状态和结果交换变成更一致的协议对象。

因此，它关注的不是某个 Agent 内部如何思考，而是不同 Agent 在系统边界上如何合作。

---

## Why It Matters｜为什么值得关注 💡

Agent 产品正在从“一个 Agent 完成所有事情”转向“多个专业 Agent 分工协作”。

这并不只是因为任务变复杂，也因为真实组织中的能力本来就分散在不同系统和团队中。财务数据、客户信息、研发流程和外部服务往往有各自的权限、接口与负责人，很难全部塞进一个 Agent 的上下文和工具列表。

更合理的方式可能是让不同 Agent 保持专业边界，再通过任务协作完成用户目标。

例如，一个项目管理 Agent 可以理解用户的整体目标，但把代码检查交给开发 Agent，把预算分析交给财务 Agent，把会议安排交给日程 Agent。主 Agent 不需要知道每项能力的内部细节，只需要理解对方能够承担什么任务、如何发起请求，以及怎样接收结果。

这让 Agent 从单体应用逐渐变成可组合服务。但同时，产品问题也从“这个 Agent 能做什么”扩展为“多个 Agent 如何形成一条用户可以理解和控制的任务链”。

A2A 值得关注，正是因为它把 Agent 间协作从概念推进到了能力描述、任务生命周期、状态更新和结果交换等更具体的系统结构。

---

## How It Works｜协作结构如何形成 ⚙️

从产品理解角度看，A2A 可以被拆成能力发现、任务委派、状态跟踪和结果交付四个环节。

### Agent Card｜让 Agent 说明自己是谁

Agent Card 是 Agent 对外提供的能力说明。

它可以包含 Agent 的身份、服务地址、支持的交互能力、安全方式和可提供的 Skills。调用方先读取这份说明，再判断对方是否适合承担当前任务。

这相当于给 Agent 建立一张可以被系统读取的“能力名片”。它让能力发现不再完全依赖人工文档或预先写死的集成关系。

从产品角度看，Agent Card 也提出了一个新问题：能力应该如何被命名和描述，才能让另一个 Agent 正确理解。描述过于宽泛会导致错误委派，描述过于技术化又难以映射到用户目标。

### Task Lifecycle｜把协作变成可跟踪任务

跨 Agent 协作通常不是一次立即返回的函数调用。

有些任务需要查询多个系统、等待外部结果、要求用户补充信息，或者在执行中再次授权。因此，A2A 把任务视为具有状态的对象，支持进行中、需要输入、需要授权、完成、失败或取消等不同阶段。

这让调用方不必把长任务当成一次无法观察的等待，而可以持续接收状态变化，并决定什么时候展示进度、请求用户介入或停止任务。

### Messages and Artifacts｜区分沟通与交付物

Agent 之间交换的不只是聊天文本。

过程中的解释、问题和补充信息可以通过 Message 传递，最终结果则可能表现为文件、结构化数据或其他 Artifact。将交流过程与交付物区分开，可以让产品更清楚地知道哪些内容只是中间沟通，哪些结果可以进入后续工作流。

对用户界面而言，这也意味着 Agent 协作不应全部压缩进聊天记录。任务状态、待确认信息和最终交付物需要采用不同的呈现方式。

### Opaque Execution｜保留系统内部边界

A2A 并不要求一个 Agent 暴露自己的内部记忆、工具或推理过程。

协作建立在公开能力、输入输出和任务状态之上。这种不透明执行有利于保持系统边界，但也会带来信任问题：调用方如何判断对方能力是否可靠，结果是否来自正确的数据，以及失败时责任如何定位。

因此，协议层解决“如何连接”之后，产品仍然需要补充身份验证、权限控制、结果验证、日志和服务质量等机制。

## Product Boundary｜A2A 与 MCP 的边界 🧭

A2A 和 MCP 经常同时出现，但它们解决的不是同一个问题。

MCP 主要标准化 Agent 或模型如何连接工具、API、数据源和外部资源。它更像是 Agent 获取某项具体能力的连接层。

A2A 则关注独立 Agent 之间如何以协作方身份交流。它不仅传递调用参数，还涉及能力发现、任务委派、长任务状态、消息、交付物和跨系统协作关系。

可以把两者放在同一条任务链中理解：主 Agent 通过 A2A 把研究任务委派给另一个专业 Agent；专业 Agent 在执行过程中，再通过 MCP 调用搜索、数据库或文档工具；完成后，它通过 A2A 返回任务状态和研究结果。

这种关系说明，MCP 更接近“Agent 如何使用能力”，A2A 更接近“Agent 如何与另一个 Agent 合作”。

仓库中已有的 MCP Servers 讨论工具和数据源连接，MCP Authorization 讨论授权边界。A2A 文章只会简要引用这些基础，不重复展开工具接入，而把重点放在跨 Agent 的任务关系和产品体验上。

## Use Case｜适用场景 📌

A2A 更适合能力分布在多个独立系统中的复杂任务，而不是为了让一个简单 Agent 看起来更复杂。

在企业内部，不同部门可以维护自己的专业 Agent。销售 Agent 可以请求合同 Agent 检查条款，再请求财务 Agent 评估付款条件。每个 Agent 保留自己的数据权限和业务逻辑，主任务仍然可以向用户呈现为一条连续流程。

在跨平台服务中，一个个人助理 Agent 可以发现并调用旅行、支付、日程或客户服务 Agent。用户表达的是完整目标，系统通过多个专业服务完成不同环节。

在长任务场景中，研究、数据处理或软件开发可能持续较长时间。A2A 的任务状态、流式更新和异步通知可以帮助上层产品持续显示进展，而不是让用户面对一个长时间没有反馈的对话框。

但并非所有多步骤任务都需要 A2A。如果所有能力都在同一个应用内，并且可以通过稳定的工作流或工具调用完成，引入跨 Agent 协议可能只会增加身份、网络、状态和错误处理成本。

是否采用 A2A，关键不在于任务步骤有多少，而在于协作对象是否真正拥有独立边界。

## Product Insight｜产品观察 🔍

A2A 对 AI 产品最大的启发，是用户目标与系统能力之间可能出现新的中间层：Agent 网络。

过去，产品通常直接连接功能模块或第三方 API。未来，产品可能先选择一个能够承担目标的 Agent，再由这个 Agent 决定如何使用自己的工具和流程。

这会让能力组合更灵活，但也会降低执行过程的可见性。用户不一定知道任务被交给了谁、哪些数据被传递、当前卡在哪个系统，以及最终结果由哪个 Agent 负责。

因此，Agent 间互操作不能只停留在后台协议。产品需要提供清楚的协作体验：展示参与任务的 Agent、说明委派原因、标记数据范围、统一不同系统的状态，并在关键动作前保留用户确认。

另一个重要变化是服务评价方式。用户可能不只评价主 Agent，还需要判断某个专业 Agent 的能力、可靠性和响应时间。未来的 Agent 生态可能需要类似服务目录的发现机制，以及围绕身份、权限、质量和责任建立的信任体系。

这也提醒我们，多 Agent 并不天然带来更好的产品。每增加一个协作方，都会增加状态同步、错误传播、权限管理和结果整合成本。只有当专业分工带来的价值大于这些成本时，跨 Agent 协作才有意义。

## Reusable Value｜可复用价值 🔁

在设计跨 Agent 产品时，可以从四个问题开始。

第一，为什么这个任务需要交给独立 Agent，而不是由现有工作流或工具完成。这个判断决定系统是否真的需要跨边界协作。

第二，委派前需要知道对方什么信息。能力范围、输入要求、预期结果、安全方式和服务状态都应足够清楚，避免主 Agent 仅凭模糊描述选择协作方。

第三，任务过程中用户需要看到哪些状态。正在执行、等待输入、需要授权、失败和完成不能只作为后台字段存在，还需要转换成可理解的界面反馈。

第四，结果如何被验证和继续使用。另一个 Agent 返回的内容不应因为来自“专业 Agent”就被默认信任，关键结果仍需来源、结构校验或人工确认。

这套思路可以用于企业 Agent 平台、专业 Agent 市场、跨组织自动化和多服务个人助理。它帮助产品团队把多 Agent 从一张架构图，转化为可发现、可委派、可追踪和可治理的真实协作关系。

A2A 展示的并不是一个“所有 Agent 自动合作”的简单未来，而是一套新的产品边界。协议可以建立共同语言，真正决定体验的仍然是任务如何被分配、状态如何被解释、数据如何被控制，以及用户是否始终知道系统正在替自己做什么。
