# Computer Use｜让 Agent 从浏览器操作走向图形界面任务执行 🖥️

- Source｜来源：Anthropic Computer Use Docs
- Type｜类型：AI 产品工具调用观察 / Computer Use / GUI Agent / Agent Safety
- Link｜链接：https://docs.anthropic.com/en/docs/build-with-claude/computer-use

当 AI 需要完成真实任务时，最大的限制往往不是模型不会理解需求，而是很多软件没有为 AI 提供可直接调用的 API。

用户每天使用的后台系统、桌面软件和网页工具，仍然主要通过按钮、输入框、菜单和弹窗完成操作。即使模型知道下一步应该做什么，也需要一种方式进入这些图形界面。

Computer Use 提供了一种直接路径：让 Agent 观察屏幕状态，并通过鼠标和键盘等操作与界面交互。

从产品视角看，它代表的不只是工具能力升级，而是 Agent 行动边界的变化。过去，AI 主要在结构化工具允许的范围内执行任务；现在，它开始尝试使用原本为人设计的软件界面。

这会扩大 Agent 可以进入的场景，也会把界面理解、执行可靠性、用户接管和安全控制变成新的产品问题。

---

## What It Is｜它是什么 🧩

Computer Use 可以理解为面向图形用户界面的 Agent 工具。

系统把当前屏幕或应用状态提供给模型，模型根据用户目标判断下一步动作，例如移动指针、点击、输入文本、滚动或使用键盘。执行环境完成动作后，再把新的界面状态返回给模型，形成持续的观察与行动循环。

它与传统 Tool Calling 的区别，在于工具接口不再提前把业务能力整理成清晰函数。

一个结构化日历工具可能直接提供“创建日程”的名称、参数和返回结果；Computer Use 面对的则是日历应用本身。Agent 需要先理解界面中哪个按钮代表新建、哪些字段需要填写、提交后页面是否出现成功状态。

因此，Computer Use 并不是“多了一个鼠标工具”这么简单。它让 Agent 承担了过去由用户完成的界面识别、操作规划和结果确认。

---

## Why It Matters｜为什么值得关注 💡

API 和 MCP 可以让 AI 通过结构化方式连接外部系统，但真实工作环境并不会立即为每个软件提供适合 Agent 的接口。

很多旧系统、内部后台和桌面应用只有图形界面。有些操作虽然可以通过 API 完成，但接入成本较高，或者用户只希望 AI 暂时协助完成一段流程，而不愿先建设完整集成。

Computer Use 降低了这类场景的连接门槛。只要用户能够通过界面完成任务，Agent 理论上就有机会沿着相似路径参与操作。

但这种通用性也带来代价。结构化工具会明确告诉模型有哪些动作和参数，图形界面却充满视觉变化、隐藏状态和模糊反馈。按钮位置可能改变，弹窗可能遮挡内容，网络延迟可能让界面停留在中间状态，相似控件也可能导致误操作。

所以，Computer Use 的核心挑战不只是“能否点击”，而是 Agent 能否理解当前状态、判断动作是否成功，并在不确定时停止或向用户求助。

这使它成为观察 Agent 产品成熟度的典型场景：能力越通用，产品越需要为可靠性和控制权提供清晰设计。

## How It Works｜它如何工作 ⚙️

从产品体验看，Computer Use 的执行可以拆成观察、行动和验证三个连续阶段。

### Observe｜观察界面

Agent 首先需要理解当前屏幕上出现了什么，以及这些视觉信息与用户目标有什么关系。

它不仅要识别文字和控件，还要判断应用状态。例如，一个按钮是可点击还是禁用，一个弹窗是提示还是确认，一段内容是最终结果还是加载中的占位状态。

如果系统只看到局部截图，可能缺少页面上下文；如果一次提供过多视觉信息，又会增加判断难度。因此，观察界面的产品设计需要处理视野范围、窗口焦点、滚动位置和敏感信息暴露等问题。

### Act｜执行动作

完成判断后，Agent 通过鼠标或键盘执行操作。

这里的关键不是单个动作能否完成，而是动作是否与用户授权的目标一致。输入文字通常比提交表单风险低，打开页面通常比删除记录风险低。

产品需要根据动作影响设置不同控制方式。低风险探索可以自动进行，高风险动作则应在执行前展示对象、范围和后果，并保留用户确认。

### Verify｜验证结果

动作完成后，Agent 不能默认任务已经成功。它需要重新观察界面，寻找状态变化和结果反馈。

点击“保存”后是否出现成功提示？页面跳转是否到达正确对象？输入内容是否被截断？操作失败后是否出现新的错误信息？

验证是 Computer Use 与简单自动点击之间的重要区别。成熟的 Agent 不是按照预设步骤机械运行，而是在每一步之后根据反馈更新判断。

## Use Case｜适用场景 📌

Computer Use 更适合需要进入现有软件、任务路径相对清楚、但缺少结构化接口的场景。

在重复性后台操作中，Agent 可以帮助用户打开系统、查找记录、整理信息和填写草稿。最终提交仍可以留给用户确认，从而在减少操作负担的同时保留责任边界。

在跨工具工作流中，Agent 可以从一个界面读取信息，再进入另一个应用完成后续步骤。这类任务过去需要多个 API 集成，Computer Use 可以先用于验证流程价值，再决定是否建设更稳定的专用连接。

在软件测试和质量检查中，Agent 可以按照用户路径访问页面、执行操作并记录异常。它不仅检查页面是否能打开，还可以观察实际交互中出现的提示、布局和状态变化。

但它不适合默认用于所有任务。涉及支付、删除、权限变更、敏感数据或不可逆操作时，单纯依赖视觉判断风险较高。产品必须缩小执行范围，增加确认和审计，而不是把“通用操作能力”理解成无限授权。

## Product Insight｜产品观察 🔍

Computer Use 最值得关注的地方，是它把原本面向人的界面变成了 Agent 的行动空间。

现有的 Playwright MCP 文章讨论了 AI 如何通过浏览器自动化进入网页任务；Computer Use 的独立重点在于，它不依赖网页内部的结构化可访问性，而是进一步面对任意视觉界面。

这意味着 Agent 可以覆盖更多软件，也意味着执行过程更容易受到页面变化和视觉误判影响。

现有的 Tool Calling UX 文章强调工具选择、参数、权限和反馈。Computer Use 在这个基础上增加了一层：工具本身没有直接返回清晰业务结果，Agent 必须从界面变化中推断执行是否成功。

因此，GUI Agent 产品需要把“当前看到什么、准备做什么、做完后确认了什么”变成可理解的执行记录。

用户不需要观看每一次鼠标移动，但应该能在关键节点知道 Agent 正在操作哪个应用、影响哪个对象，以及为什么需要确认。出现异常时，用户还应能够暂停、接管或回到上一个安全状态。

从这个角度看，Computer Use 的产品体验不应模仿一个完全隐形的自动化脚本，而应更接近一个可以被监督的数字协作者。

## Reusable Value｜可复用价值 🔁

设计 Computer Use 类产品时，可以先把任务拆成三个层次：只读观察、可撤回编辑和不可逆执行。

只读观察包括浏览页面、读取状态和收集信息，可以给予更高自动化程度。可撤回编辑包括填写草稿、调整内容和暂存设置，需要向用户展示修改结果。不可逆执行包括发送、删除、支付和发布，应设置明确确认门槛。

还可以为每个任务定义成功证据。Agent 不应只记录“点击了保存”，而要记录页面出现了什么结果、对象状态是否变化，以及是否存在需要人工检查的异常。

这套框架也适用于评价 GUI Agent。判断重点不只是任务是否最终完成，还包括中途发生变化时能否恢复、用户是否保持控制、敏感信息是否被限制在必要范围内。

## Productization Direction｜产品化方向 🚀

### Supervised Task Runner｜可监督任务执行器

产品可以把 Computer Use 组织成带有目标、步骤和检查点的任务执行器。用户看到的是当前目标、已完成动作和等待确认的关键节点，而不是一段难以理解的自动化日志。

### Interface Action Policy｜界面操作策略

团队可以为不同应用和动作建立风险策略。例如允许自动搜索和读取，但发送、删除和权限变更必须确认。策略应根据影响而不是按钮名称判断风险。

### Replay and Recovery｜回放与恢复

GUI 操作容易受到环境变化影响。产品可以保存关键截图、动作记录和结果状态，帮助用户理解问题发生在哪里，并从最近的安全检查点继续任务。

Computer Use 展示了 Agent 从结构化工具走向开放软件环境的可能性。

它真正的产品价值，不是让 AI 像人一样点击得更快，而是让用户不必亲自完成每个界面步骤，同时仍然能够理解、限制和接管整个执行过程。
