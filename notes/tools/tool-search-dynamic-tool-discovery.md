# Tool Search｜让 Agent 按需发现和加载工具 🔎

- Source｜来源：Anthropic Claude Platform Docs
- Type｜类型：AI 产品方法观察 / Tool Search / Dynamic Tool Discovery / Agent Tooling
- Link｜链接：https://platform.claude.com/docs/en/agents-and-tools/tool-use/tool-search-tool

当一个 Agent 只连接少量工具时，系统可以在每次请求中直接提供全部工具定义。

但随着产品接入文件、浏览器、数据库、代码仓库、业务系统和多个 MCP Server，工具目录会迅速扩大。模型需要在越来越多的名称、描述和参数之间判断应该调用什么，不相关工具也会持续占用上下文。

Tool Search 提供了另一种方式：不再默认加载全部工具，而是先根据当前任务检索可能相关的工具，再把选中的定义提供给模型。

从产品视角看，它解决的不只是上下文长度问题，而是 Agent 能力规模扩大后，工具如何被组织、发现和使用的问题。

> 当工具从几个按钮变成一个能力目录，工具发现本身就会成为产品基础设施。

---

## What It Is｜它是什么 🧩

Tool Search 可以理解为面向 Agent 的动态工具检索机制。

传统 Tool Calling 通常会把可用工具的名称、用途和参数结构全部放进模型上下文。模型阅读这些定义后，判断是否调用某个工具，以及应该提供哪些参数。

当工具数量不多时，这种方式简单直接。但在大型工具目录中，大量无关定义会增加上下文负担，也可能让相似工具之间更难区分。

根据 Anthropic 的官方文档，Tool Search 允许系统将部分工具标记为按需加载。模型先使用搜索能力查找相关工具，系统再返回匹配的工具定义，随后模型才能调用这些工具。

这里被搜索的不是网页内容或业务数据，而是“当前有哪些可执行能力”。搜索结果也不是最终答案，而是下一步任务执行所需的工具入口。

---

## Why It Matters｜为什么值得关注 💡

Agent 产品的能力边界越来越依赖工具。

模型本身可以理解和生成信息，但要读取文件、查询数据库、发送消息或操作业务系统，就需要连接外部工具。随着工具持续增加，一个新问题会出现：Agent 是否能在正确时间找到正确能力。

如果所有工具始终加载，产品会面临三个压力。

首先是上下文压力。每个工具都需要名称、描述和参数说明，这些内容会和用户任务、历史消息及其他资料争夺上下文空间。

其次是选择压力。多个工具可能名称接近、用途重叠，模型需要在大量候选项中完成路由，错误选择会让任务偏离目标。

最后是维护压力。工具变化后，目录、描述和参数需要持续更新，产品也需要知道哪些工具应该始终可见，哪些只在特定场景出现。

Tool Search 的价值，是把一次性展示全部能力，改成围绕当前任务逐步发现能力。

这和传统软件中的全量菜单与按需搜索有相似之处：功能越多，信息架构和检索质量越重要。

## How It Works｜它如何工作 ⚙️

从产品结构看，Tool Search 可以拆成工具目录、检索路由和动态加载三个部分。

### Tool Catalog｜工具目录

每个工具需要清楚的名称、用途描述和参数结构。

这些信息不只是给开发者看的接口说明，也会成为搜索和选择工具的基础。如果描述过于宽泛，多个工具可能同时匹配；如果缺少场景和边界，Agent 即使找到工具，也可能错误理解它的用途。

因此，工具元信息开始承担产品导航的角色。团队需要像设计功能分类一样设计工具目录。

### Retrieval｜检索相关能力

当用户提出目标后，系统先根据任务检索相关工具。

例如，用户希望整理某个代码仓库的问题记录，系统可能优先发现 GitHub 查询工具，而不必同时加载日历、邮件和数据库工具。

检索可以减少候选范围，但它本身也可能出错。用户表达模糊、工具描述相似或目录分类不清时，真正需要的工具可能没有被找到。

这意味着 Tool Search 需要被评估，而不能被视为一个透明且必然正确的中间步骤。

### Dynamic Loading｜动态加载

检索到相关工具后，系统再把完整定义提供给模型。

这种方式让核心工具与延迟加载工具形成分层。高频、基础、低风险的能力可以始终存在；专业、低频或上下文较大的工具，则根据任务需要加载。

分层的价值不是隐藏功能，而是让 Agent 在每一步只面对更相关的能力集合。

## Use Case｜适用场景 📌

Tool Search 更适合工具数量较多、来源分散或能力经常变化的 Agent 产品。

企业内部 Agent 可能连接知识库、工单、CRM、数据分析和协作平台。不同角色需要不同工具，如果全部加载，不仅增加上下文，也会模糊权限和任务边界。

开发类 Agent 可能访问代码仓库、终端、测试、部署和文档工具。用户提出不同目标时，真正需要的工具组合并不相同。

MCP 生态也会放大工具发现需求。一个 MCP Server 可以暴露多个工具，用户又可能同时连接多个 Server。连接能力解决了“工具如何接入”，Tool Search 则进一步解决“任务发生时如何找到相关工具”。

如果产品只有少量稳定工具，直接提供全部定义通常更简单。动态检索会增加额外步骤和延迟，不应为了追求 Agent 形式而引入。

## Product Insight｜产品观察 🔍

Tool Search 最值得关注的地方，是它让“能力发现”成为 Agent 产品中的独立环节。

传统软件通常把功能展示给用户，由用户点击菜单或按钮。Agent 产品允许用户直接描述目标，系统需要在后台完成意图理解、能力匹配和工具调用。

界面看起来可能只是一个输入框，但背后已经存在一套能力导航系统。

这也说明 Tool Search 与 Tool Calling UX 解决的是不同问题。

Tool Search 关注 Agent 在运行时如何找到工具；Tool Calling UX 关注工具被选择后，用户如何理解调用目的、确认敏感操作并查看执行结果。找到工具并不代表可以直接使用，更不代表执行一定可靠。

Tool Search 也不同于 Agent Skills。Skill 封装的是某类任务的方法、说明、资源和可复用能力；Tool Search 更聚焦可执行工具定义的检索与加载。一个 Skill 可以指导 Agent 完成任务，并在过程中使用 Tool Search 找到需要的外部工具。

把这些对象区分开，能避免将 Prompt、Skill、Tool 和 Agent 全部混成模糊的“能力”概念。

## Reusable Value｜可复用价值 🔁

设计大型工具目录时，可以先将工具分成三类：始终需要的核心工具、特定场景才需要的专业工具，以及需要额外授权的敏感工具。

核心工具强调低延迟和稳定可用，专业工具适合按需检索，敏感工具则需要在被发现之后继续执行授权与确认流程。

团队还需要为工具描述建立统一规范。除了“这个工具能做什么”，描述还应说明适用场景、输入要求和不应该使用的边界。清楚的元信息既帮助模型检索，也方便团队维护能力目录。

评估时不能只看工具是否最终被调用，还应观察检索是否找到了正确候选、是否加载了过多无关工具、额外搜索步骤增加了多少延迟，以及用户任务完成率是否改善。

这些指标能帮助团队判断问题究竟出在工具不存在、检索没有找到，还是调用过程本身失败。

## Productization Direction｜产品化方向 🚀

### Agent Tool Registry｜Agent 工具注册中心

AI 应用平台可以建立统一工具注册中心，记录每个工具的来源、用途、参数、权限、版本和适用范围。

它不仅服务开发接入，也让产品团队理解当前 Agent 实际拥有怎样的行动能力。

### Tool Discovery Analytics｜工具发现分析

平台可以展示用户任务对应的搜索词、候选工具、最终选择和失败原因。

当某个工具经常应该被使用却没有被发现时，团队可以修改描述、分类或路由规则，而不是只调整模型 Prompt。

### Role-based Tool Spaces｜角色化工具空间

不同用户角色可以拥有不同的工具目录。

产品经理、开发者和运营人员即使使用同一个 Agent，也可以在各自权限与任务范围内发现不同能力。这能同时减少选择噪音，并让工具访问边界更清楚。

Tool Search 体现了 Agent 产品从“连接工具”走向“管理工具规模”的变化。

真正成熟的工具系统，不是让 Agent 看到尽可能多的能力，而是让它在正确场景找到恰当工具，并在清楚的权限和反馈机制下完成任务。
