# Agent Skills｜把专业能力封装成可发现、可组合的 AI 能力单元 🧩

- Source｜来源：Anthropic Agent Skills Docs
- Type｜类型：AI 产品方法观察 / Agent Skills / Capability Packaging / Progressive Disclosure
- Link｜链接：https://docs.anthropic.com/en/docs/agents-and-tools/agent-skills/overview

很多 AI 产品会从 Prompt 开始构建能力。

团队把任务说明、背景知识和输出要求写进一段提示词，让模型完成某个具体工作。这种方式适合快速验证，但当任务逐渐增多，Prompt 会变得越来越长，用户也需要反复补充相同信息。

Agent Skills 提供了另一种思路：把某类任务需要的说明、流程、参考资料、模板和脚本封装成一个可以被发现、按需加载和重复使用的能力单元。

从产品视角看，Skill 的价值不只是减少 Prompt 重复，而是让 AI 产品开始拥有可管理的“能力资产”。用户提出目标后，系统不必把所有知识一次性塞进上下文，而是先判断需要什么能力，再加载完成当前任务所需的内容。

这意味着 AI 产品的入口正在从“用户如何写好一句 Prompt”，逐渐转向“系统如何为用户找到合适的能力”。

---

## What It Is｜它是什么 🧠

Agent Skill 可以理解为一个围绕特定任务组织起来的能力包。

根据 Anthropic 的 Agent Skills 文档，一个 Skill 可以包含用于发现能力的元信息、被触发后才读取的具体说明，以及按任务需要继续访问的脚本、模板和参考材料。它不是一段固定回复，也不等于某个独立 Agent，而是为 Agent 提供某类专业工作方式。

例如，一个文档审阅 Skill 不只是写着“请帮我检查文档”。它还可以说明审阅标准、执行顺序、输出结构，并附带术语表、检查清单或验证脚本。Agent 在遇到匹配任务时调用这些内容，按照更稳定的路径推进工作。

这与现有的产品经理 Skills 文章形成了上下层关系。那篇文章关注需求拆解、竞品分析和 PRD 梳理等产品任务如何被封装；这一篇关注的是更基础的问题：一个 Skill 为什么能被识别、何时应该加载，以及如何把说明、资源和确定性工具组织成同一个能力单元。

---

## Why It Matters｜为什么值得关注 💡

通用 Agent 面临一个长期矛盾：它需要知道足够多的信息才能完成专业任务，但又不能把所有规则、资料和案例长期放进上下文。

如果系统把大量说明全部写进 System Prompt，上下文会变得臃肿。真正需要的规则容易被无关内容淹没，维护时也很难判断某次修改会影响哪些任务。

如果完全依赖用户临时描述，专业流程又会变得不稳定。不同用户对同一个任务的表达方式不同，模型可能遗漏步骤，输出质量也过度依赖用户是否“会提问”。

Skills 的意义在于，它在通用 Agent 和具体任务之间增加了一层能力组织。

用户仍然可以用自然语言表达目标，但系统通过 Skill 的名称和描述判断应该启用什么能力。只有被触发的 Skill 才进一步提供完整说明，需要时再读取额外材料或运行脚本。

这种方式把“如何完成任务”的责任从用户输入转移到产品内部。用户不必每次重建专业流程，团队也可以持续改进同一个能力单元，而不是在不同 Prompt 中重复维护相似规则。

## How It Works｜它如何工作 ⚙️

从产品结构看，一个 Skill 可以拆成发现、执行和资源调用三个层次。

### Discovery｜发现能力

系统首先看到的是 Skill 的名称和描述。描述不仅要说明它能做什么，也要说明在什么情况下应该使用。

这一层看起来只是元信息，实际上决定了能力入口是否有效。如果描述过于宽泛，Skill 可能在无关任务中被错误触发；如果描述过于狭窄，用户明明需要这个能力，系统却无法发现它。

因此，Skill Discovery 本质上是一种新的信息架构。产品需要设计的不只是功能名称，还包括用户意图与能力之间的匹配关系。

### Instructions｜加载工作方法

当 Skill 被触发后，Agent 才读取完整说明。这里可以定义任务步骤、判断规则、输入要求、输出标准和异常处理方式。

这种按需加载也被称为渐进披露。系统先保留轻量的能力目录，只有进入具体任务时才展开详细内容，从而避免无关说明持续占用上下文。

从用户体验看，渐进披露意味着产品可以同时拥有较多能力，又不必把所有能力平铺在一个复杂界面中。用户表达目标，系统负责在后台选择和展开相应工作方式。

### Resources and Scripts｜调用资源与脚本

有些任务只靠自然语言说明不够稳定。Skill 还可以附带模板、参考资料和可执行脚本。

说明适合处理需要判断和适应的步骤，模板适合统一输出结构，脚本则适合计算、校验和文件处理等需要确定性的动作。把这些内容放在同一个能力包中，可以让模型推理与确定性工具形成分工。

这也是 Skill 和普通 Prompt 的重要差异：Prompt 主要提供文本指令，Skill 则可以同时组织工作方法、知识资源和执行工具。

## Use Case｜适用场景 📌

Skills 更适合重复出现、具有相对稳定方法、同时又需要一定上下文判断的任务。

在产品工作中，需求澄清、用户反馈归类、PRD 检查和项目复盘都可以形成 Skill。每次任务的具体内容不同，但分析步骤、质量标准和输出结构具有复用价值。

在内容与文档场景中，品牌语气、编辑规则、引用要求和交付格式可以被封装起来。用户不必在每次请求中重复说明，Agent 也能更稳定地遵循同一套标准。

在团队场景中，Skills 还可以沉淀组织内部的方法和知识。例如，把发布检查、数据分析规范或客户支持流程组织成项目级能力，让不同成员调用同一套工作方式。

但并不是所有内容都适合做成 Skill。一次性的背景信息、频繁变化的数据和只有单步回答的简单问题，未必需要额外封装。Skill 的价值来自复用和稳定性，而不是把所有 Prompt 都改造成文件夹。

## Product Insight｜产品观察 🔍

Agent Skills 最值得关注的地方，是它让 AI 产品的能力从“隐藏在 Prompt 里的文本”变成了可以被组织、发现和维护的产品对象。

传统软件通常通过菜单、按钮和页面展示功能。Agent 产品允许用户直接描述目标，因此功能入口不一定全部可见，但能力仍然需要清楚的边界。

Skill 的描述承担了类似功能导航的作用。它告诉系统什么任务应该进入这个能力，而 Skill 内部的说明则定义了任务如何执行。对用户来说，界面可能仍然只是一个输入框；对产品内部来说，背后已经形成一套能力路由。

这也带来新的产品问题。用户是否知道当前调用了哪个 Skill？多个 Skills 同时匹配时如何选择？Skill 更新后会不会改变既有任务结果？团队如何判断一个 Skill 真的提升了完成质量，而不是增加了更多规则？

因此，Skills 产品化不能只解决“如何创建”，还需要解决发现准确性、版本管理、权限边界和效果评估。

另一个重要观察是，Skill 并不取代 Prompt、Tool 或 Agent。Prompt 仍然负责表达当前任务，Tool 提供外部行动能力，Agent 负责围绕目标持续推进，而 Skill 更像把某类任务的专业工作方式提供给 Agent。

当这些对象被清楚区分后，AI 产品的能力结构会更容易理解和迭代。

## Reusable Value｜可复用价值 🔁

设计 Skill 时，可以先判断任务是否具有稳定的输入、过程和质量标准，再决定哪些内容属于说明，哪些内容应该成为参考资源或脚本。

一个可复用的 Skill 至少需要回答几个问题：它解决什么任务，什么时候应该触发，需要用户提供哪些信息，完成过程有哪些关键判断，最终结果如何检查。

还需要关注能力边界。一个 Skill 如果同时承担需求研究、方案设计、数据分析和文档输出，内部规则会迅速变得复杂，也不利于与其他能力组合。更合理的方式是让每个 Skill 围绕一个清晰任务负责，再由 Agent 根据目标组合多个能力。

这套方法也可以用于分析现有 AI 产品。观察一个产品时，不只看它有哪些按钮，还可以判断它是否已经把高频任务沉淀为明确能力，用户是否能自然触发这些能力，以及能力的执行过程是否足够稳定。

## Productization Direction｜产品化方向 🚀

### Skill Library｜能力库

AI 产品可以把 Skills 组织成可搜索、可预览的能力库。用户能够理解每个 Skill 适合什么任务、需要什么输入、会使用哪些工具，并根据自己的工作方式启用或关闭。

### Skill Composer｜能力组合器

复杂任务通常需要多个能力协作。产品可以让用户把研究、分析、生成和验证类 Skills 组合成一条任务路径，同时保留 Agent 根据实际情况调整步骤的空间。

### Skill Governance｜能力治理

当团队开始共享 Skills，版本、来源和权限会变得重要。产品需要记录谁创建了 Skill、最近修改了什么、它可以访问哪些数据和工具，以及某个版本是否经过验证。

Skills 体现了 AI 产品从“一次性对话”走向“能力积累”的变化。

真正有价值的能力库，不是拥有最多的 Skill，而是让用户表达目标后，系统能够找到恰当能力、加载必要信息，并通过稳定方法把任务推进到可用结果。
