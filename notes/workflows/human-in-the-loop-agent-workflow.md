# Human-in-the-loop｜Agent 工作流中的人类介入机制 🧑‍💻

- Source｜来源：LangChain Docs / Cloudflare Agents Docs
- Type｜类型：AI 产品工作流观察 / Human-in-the-loop / Agent Workflow / AI UX / Risk Control
- Link｜链接：https://docs.langchain.com/oss/python/langchain/human-in-the-loop

Human-in-the-loop，通常可以翻译为“人类介入机制”或“人在回路中”。

在 AI Agent 和 AI Workflow 场景里，它不是简单地让人“最后检查一下”，而是在 AI 执行任务的关键节点中，设计人工确认、人工修正、人工接管和人工反馈机制。

如果说 Tool Calling UX 关注的是：

> AI 调用工具时，用户如何理解和控制过程。

那么 Human-in-the-loop 更进一步关注：

> 当 AI 开始执行复杂任务时，哪些步骤可以自动完成，哪些步骤必须让人介入。

从产品视角看，Human-in-the-loop 代表了 Agent 产品落地中的一个关键问题：

> AI 产品不是越自动越好，而是要在自动化效率和人类控制之间设计边界。

---

## What It Is｜它是什么

Human-in-the-loop 是指在 AI 系统或 Agent 工作流中，把人类判断作为流程的一部分。

它通常出现在以下场景：

- AI 准备执行高风险动作前
- AI 不确定下一步该怎么做时
- 工具调用结果需要用户确认时
- 多步骤任务进入关键节点时
- AI 输出需要人工审核时
- 自动化流程失败后需要人工接管时

在普通 Chatbot 中，用户主要通过输入 Prompt 来影响 AI。

但在 Agent Workflow 中，用户不仅是“提问者”，还可能是：

- 决策者
- 审核者
- 修正者
- 授权者
- 接管者
- 反馈提供者

因此，Human-in-the-loop 的本质不是降低 AI 自动化能力，而是让 AI 工作流在关键位置保留人类判断权。

---

## Why It Matters｜为什么值得关注 💡

Human-in-the-loop 值得关注，是因为 Agent 产品一旦进入真实任务，就会遇到自动化边界问题。

一个 Agent 可以自动完成很多事情，例如：

- 搜索资料
- 总结文档
- 调用工具
- 生成草稿
- 整理表格
- 修改文件
- 创建任务
- 提交代码
- 发送消息

但这些动作的风险等级不同。

低风险任务可以自动化；高风险任务如果完全自动执行，就可能带来错误、误操作、权限越界或用户信任问题。

例如：

- 总结公开网页，风险较低
- 读取私人文件，风险中等
- 修改仓库文件，风险较高
- 删除内容或发送邮件，风险更高

LangChain 的 Human-in-the-loop middleware 强调，可以在模型提出某些工具调用时暂停执行，并等待人工决策。Cloudflare Agents 文档中也把 workflow approval 和 MCP elicitation 作为两类 Human-in-the-loop 模式：前者适合多步骤流程中的审批门槛，后者适合 MCP Server 请求结构化用户输入。

这些实践说明，人类介入不是临时补丁，而是 Agent 工作流本身需要设计的一部分。

---

## How It Works｜它如何工作 ⚙️

Human-in-the-loop 可以从几个典型动作理解。

### Approve｜确认执行

当 AI 准备执行高风险动作时，系统暂停执行，让用户确认是否继续。

例如：

- 发送邮件前确认
- 删除文件前确认
- 提交代码前确认
- 修改数据库前确认
- 调用付费 API 前确认

Approve 解决的是：

> AI 可以提出动作，但最终执行权仍然保留给用户。

这类机制尤其适合行动型 AI 产品。

因为一旦 AI 能修改真实系统状态，用户就需要清楚知道它将做什么，并有机会拒绝。

### Edit｜人工修改

Edit 是指用户不只是选择同意或拒绝，还可以修改 AI 生成的中间结果。

例如：

- 修改邮件草稿
- 调整报告结构
- 编辑代码 diff
- 修改任务计划
- 调整工具调用参数

这种机制适合 AI 输出接近可用，但仍需要人类判断和微调的场景。

它的价值在于：

> 用户不必推翻 AI 的工作，而是可以在 AI 结果基础上继续修正。

### Reject｜拒绝或中止

Reject 是指用户可以拒绝 AI 的某个动作，或者中止当前任务。

适合的场景包括：

- AI 选择了错误工具
- AI 理解错了任务目标
- AI 准备修改错误文件
- AI 生成了不合适内容
- AI 要执行用户不认可的操作

Reject 解决的是：

> 用户可以阻止 AI 继续沿着错误方向执行。

如果用户只能被动等待 Agent 执行完成，而不能中途阻止，产品会缺乏控制感。

### Clarify｜补充信息

Clarify 是指 AI 在信息不足时主动向用户提问。

例如：

- “你希望修改哪一个文件？”
- “这封邮件要发给谁？”
- “这个操作是否只影响当前分支？”
- “你希望我优先参考哪个资料源？”

这类介入机制适合任务边界不清晰的场景。

它的价值在于：

> AI 不确定时不应该硬做，而应该把关键不确定性抛回给用户确认。

## Product Insight｜产品观察 🔍

Human-in-the-loop 的核心价值，不是给 AI 加一个“人工审核”按钮，而是重新设计人和 Agent 在工作流中的分工。

很多 Agent 产品容易陷入一个误区：

> 自动化越多，产品越先进。

但真实情况是，用户真正需要的不是“完全自动”，而是“可托付”。

可托付意味着：

- 简单任务可以自动完成
- 关键节点需要明确确认
- 用户可以理解 AI 正在做什么
- 用户可以修改中间结果
- 用户可以拒绝错误动作
- 用户可以在必要时接管流程
- 执行结果可以追溯和回滚

因此，Human-in-the-loop 不是自动化的反面，而是自动化产品化的条件。

### 1. 人类介入应该按风险设计

不是所有 AI 动作都需要人工确认。

如果每一步都需要用户确认，Agent 会失去自动化价值。

但如果所有动作都默认自动执行，用户又会失去控制感。

因此，产品需要按风险等级设计介入策略。

| 风险等级 | 典型动作 | 介入方式 |
|---|---|---|
| 低风险 | 搜索、总结、计算、生成草稿 | 可自动执行，轻量反馈 |
| 中风险 | 读取私人资料、访问内部系统、调用业务数据 | 提示访问范围，必要时确认 |
| 高风险 | 发送、删除、提交、写入、支付、更新数据库 | 必须人工确认，可预览可撤销 |

这个分层和 Tool Calling UX 是连续的。

工具调用决定 AI 能做什么；Human-in-the-loop 决定 AI 做到哪一步必须停下来等人。

### 2. 人类介入点应该产品化

很多产品把 Human-in-the-loop 做成“失败后找人处理”。

但更好的方式是提前把介入点设计进工作流。

例如：

- 发送邮件前进入草稿审核
- 修改文件前展示 diff
- 提交代码前展示 commit message
- 执行数据库写入前展示影响范围
- 多步骤任务中间提供 checkpoint
- Agent 不确定时发起澄清问题

这些介入点不是异常处理，而是 Agent 产品的正常交互组成。

### 3. Human-in-the-loop 不等于用户全程盯着

人类介入并不意味着用户要每一步都参与。

更合理的状态是：

> 用户不必时刻干预，但需要在关键节点拥有监督权和接管权。

例如：

- AI 自动完成资料搜索和初步整理
- 在准备修改文件前暂停
- 用户查看修改内容
- 用户选择确认、编辑或拒绝
- AI 继续执行后续流程
- 完成后给出执行摘要

这种方式既保留自动化效率，也保留用户控制感。

## Reusable Value｜可复用价值 🔁

Human-in-the-loop 可以为 AI 产品学习和实践提供几个可复用视角。

### 1. 用介入点拆解 Agent 工作流

分析一个 Agent 产品时，可以不只看它能完成什么任务，还可以看它在哪里让人介入。

可以问：

- AI 哪些步骤可以自动执行？
- 哪些步骤需要人工确认？
- 哪些中间结果允许编辑？
- AI 不确定时是否会询问用户？
- 执行失败后是否能交还控制权？
- 用户是否能取消任务？
- 高风险动作是否能预览和回滚？

这种拆解方式比只看 Agent 功能更接近真实产品体验。

### 2. 用确认机制提升用户信任

确认机制不是为了拖慢流程，而是为了让用户知道：

- AI 准备做什么
- 会影响哪些对象
- 有哪些风险
- 是否可以撤销
- 用户是否同意继续

尤其在高风险动作中，确认机制本身就是信任机制。

例如：

- “我将修改 3 个文件，是否继续？”
- “我准备发送这封邮件，请确认收件人和正文。”
- “这次操作会删除 12 条记录，是否继续？”
- “我将提交 commit：xxx，请确认。”

这些确认文案应该具体，而不是简单地问“是否继续”。

### 3. 用可编辑中间结果减少返工

Human-in-the-loop 不应该只有“确认 / 取消”两个选项。

很多真实场景中，用户需要编辑 AI 的中间结果。

例如：

- 修改计划
- 调整语气
- 删除某一段
- 修改表格字段
- 替换引用来源
- 调整文件命名

如果产品只提供重新生成，用户会失去对结果的精细控制。

更好的方式是让用户能在中间状态上直接编辑，再让 AI 继续执行。

## Productization Direction｜产品化方向 🚀

Human-in-the-loop 可以继续延展出几个产品化方向。

### 1. Approval Gate｜审批节点

Agent Workflow 平台可以提供标准化审批节点。

用户可以在工作流中配置：

- 哪类动作需要审批
- 谁来审批
- 审批通过后继续哪一步
- 审批拒绝后如何回退
- 是否记录审批日志

这类能力适合企业级 AI Workflow。

### 2. Editable Intermediate Output｜可编辑中间结果

AI 产品可以把中间结果产品化为可编辑对象。

例如：

- 邮件草稿
- 文档草稿
- 代码 diff
- 表格结果
- 任务计划
- Workflow 节点输出

用户编辑后，Agent 可以基于修改后的结果继续执行。

这会让人机协作更自然。

### 3. Agent Checkpoint｜Agent 检查点

复杂 Agent 任务可以设置 checkpoint。

例如：

- 资料收集完成后暂停
- 初稿生成完成后暂停
- 工具调用前暂停
- 高风险动作前暂停
- 最终提交前暂停

Checkpoint 的价值是把长任务切成可验证阶段，避免 Agent 一口气跑完整个流程后才发现方向错了。

### 4. Takeover Mode｜人工接管模式

当 Agent 无法继续执行时，产品可以进入接管模式。

接管模式可以展示：

- 当前任务目标
- 已完成步骤
- 当前卡点
- 工具调用记录
- 中间结果
- 建议的下一步
- 用户可执行操作

这可以让用户快速接手，而不是重新理解整个任务。

## Product Observation｜进一步观察

Human-in-the-loop 说明了一个趋势：

> Agent 产品的成熟度，不只取决于自动化能力，也取决于它如何设计人类控制权。

早期 AI 产品常常强调“让 AI 自动完成更多事情”。

但当 Agent 真正进入工作流后，产品的关键问题会变成：

- 哪些任务应该自动化？
- 哪些任务不应该自动化？
- 哪些动作需要确认？
- 用户如何修改中间结果？
- AI 失败后如何恢复？
- 人类如何重新接管流程？
- 系统如何记录人类介入过程？

因此，Human-in-the-loop 不是一个安全附加项，而是 Agent Workflow 的核心产品机制。

一个可靠的 Agent 产品，不应该只展示它能自动完成什么，也应该清楚展示它什么时候会停下来、询问用户、等待确认或交还控制权。

这也是 Human-in-the-loop 对 AI 产品实践最重要的启发。
