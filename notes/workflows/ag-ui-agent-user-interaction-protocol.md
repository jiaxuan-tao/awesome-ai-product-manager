# AG-UI Protocol｜让 Agent 状态进入用户界面 🔄

- Source｜来源：AG-UI Official Documentation / Official GitHub Repository
- Type｜类型：AI 产品工作流观察 / Agent-User Interaction Protocol / Event-driven UI
- Link｜链接：https://docs.ag-ui.com/introduction

传统 Web 产品通常建立在一次请求、一次响应的交互模型上：用户提交操作，后端返回结果，前端完成渲染。但 Agent 任务可能持续更长时间，中间包含文本生成、工具调用、状态变化、用户输入和多轮执行。等到整个任务结束后再返回一个结果，会让用户长期面对缺少解释的等待状态。

AG-UI，即 Agent User Interaction Protocol，是一个开放、轻量、基于事件的协议，用于标准化 Agent 后端与用户界面之间的连接。它不是 Agent 框架，也不是一套现成 UI，而是定义 Agent 运行时信息如何进入前端、用户和应用状态如何回到 Agent 的交互契约。

从产品视角看，AG-UI 关注的核心问题是：当 Agent 不再只是生成一段回答，而是在持续执行任务时，界面如何知道它正在做什么、状态发生了什么变化，以及什么时候需要用户参与。

---

## What It Is｜它是什么 🧩

一个 AG-UI 交互可以理解为“输入一次运行，接收一组事件，再由用户动作开启下一次运行”。

前端通过 `RunAgentInput` 向 Agent 提交请求，其中可以包含 `threadId`、`runId`、消息记录、当前状态、上下文和前端提供的工具。Agent 开始执行后，不必等待所有工作完成再返回，而是持续发出带有明确类型的事件。前端按顺序处理这些事件，将它们转换成消息、进度、工具状态、表单或其他界面变化。

这是一种逻辑上的双向连接：用户界面把消息、状态、工具和恢复信息发送给 Agent，Agent 则通过事件流把运行过程和状态更新传回界面。AG-UI 不限定必须使用某一种传输方式，可以建立在 SSE、WebSocket、Webhook 等机制之上，并提供标准 HTTP 客户端作为参考实现。

它解决的不是“页面应该长什么样”，而是“前端和 Agent 后端如何对同一次任务形成一致理解”。如果没有这层契约，不同 Agent 框架往往会产生各自的流式格式、工具消息和状态字段，前端只能为每个后端单独编写适配逻辑。

---

## Event Flow｜从 Agent 执行到界面事件 ⚙️

AG-UI 把运行过程拆成具有类型、标识符和顺序关系的事件。前端不再只处理一个最终响应，而是维护一个随事件持续变化的任务视图。

### Run Lifecycle｜运行生命周期

一次运行以 `RUN_STARTED` 开始，并以 `RUN_FINISHED` 或 `RUN_ERROR` 结束。可选的 `STEP_STARTED` 和 `STEP_FINISHED` 可以表示检索资料、分析数据或生成草稿等阶段。

`threadId` 用于关联持续对话或任务，`runId` 用于识别其中一次具体执行。产品因此可以区分“同一个长期任务”和“这一次重新生成、继续或恢复”，并为它们展示不同状态。

生命周期事件让前端能够建立明确的界面状态机：尚未开始、正在运行、阶段完成、成功结束、发生错误或等待恢复。需要注意的是，步骤事件只是 Agent 主动暴露的可观察阶段，并不天然代表完整计划，也不保证后端一定会提供细粒度进度。

### Streaming Feedback｜流式反馈

文本消息采用 Start、Content、End 模式。`TEXT_MESSAGE_START` 创建消息，连续的 `TEXT_MESSAGE_CONTENT` 提供增量文本，`TEXT_MESSAGE_END` 表示内容完成。工具调用也采用类似结构，通过 `TOOL_CALL_START`、`TOOL_CALL_ARGS` 和 `TOOL_CALL_END` 逐步传输工具名称与参数，并可通过 `TOOL_CALL_RESULT` 返回执行结果。

这种事件流让界面能够尽早提供反馈。用户可以先看到 Agent 已经开始运行、当前进入哪个阶段、正在构造哪个工具调用，再逐步获得内容，而不必面对一个没有解释的加载动画。

但流式反馈不等于把所有后端日志直接展示给用户。产品仍需把技术事件转译成与任务有关的信息，例如“正在查询订单”或“已整理 12 条记录”，并控制更新频率，避免界面因高频事件不断跳动。协议提供运行信号，产品负责决定哪些信号值得用户注意。

## Shared State｜让前后端共享任务状态 🔁

消息只能表达对话内容，却不足以承载完整任务。一个研究 Agent 可能需要保存筛选条件、候选资料和当前阶段；一个数据助手可能需要保存指标、时间范围和用户选择；一个内容工作流则可能包含大纲、草稿和待修改段落。

AG-UI 通过 `STATE_SNAPSHOT` 和 `STATE_DELTA` 同步这些结构化状态。Snapshot 提供完整状态基线，前端收到后用它替换当前状态；Delta 使用 JSON Patch 表达增量变化，前端按顺序应用更新。`MESSAGES_SNAPSHOT` 则用于重新建立完整的消息记录。

共享状态的产品价值不只是“前端能看到 Agent 内部变量”，而是建立一个人和 Agent 都能操作的任务对象。Agent 可以更新草稿或进度，用户可以在界面中修改筛选条件、选择对象或调整内容，更新后的状态再随下一次运行交给 Agent。

状态设计需要有明确边界。适合共享的是会影响界面和后续决策的产品状态，而不是把模型内部数据全部暴露出来。团队还需要定义字段所有权、更新顺序、冲突处理和重新同步机制，否则双向修改可能产生覆盖或过期状态。

## User Interaction｜中断、输入与前端工具 🧑‍💻

Agent 与用户的协作不应被压缩成“任务开始前输入一次，任务结束后查看一次”。AG-UI 提供了中断恢复和前端工具两种重要连接方式，让用户可以在运行过程中成为任务参与者。

### Interrupt and Resume｜中断与恢复

在支持 interrupt-aware lifecycle 的实现中，Agent 可以通过 `RUN_FINISHED` 的 interrupt outcome 表示当前运行因等待用户输入而暂停。Interrupt 可以携带关联 ID、原因、面向用户的说明、响应 Schema，以及可选的工具调用 ID。

前端根据这些信息渲染确认、选择或结构化输入界面。用户完成操作后，客户端在同一个 thread 中发起新的运行，并通过 `resume` 提交对应 interrupt 的处理结果。为了支持恢复，Agent 应在中断结束前发出所需的状态和消息快照。

这里的“中断”是一次运行以等待输入的结果结束，再由新运行继续，并不等同于用户点击“停止生成”时对当前网络流的取消。产品文案和状态模型应区分主动停止、运行失败、等待输入和正常完成，不能统一显示为“任务结束”。

AG-UI 定义的是中断如何被表达和恢复，不负责决定哪些业务节点必须人工介入。是否需要确认、允许哪些修改，以及拒绝后走哪条路径，仍然属于 Human-in-the-loop 工作流设计。

### Frontend Tools｜前端工具调用

AG-UI 区分后端已有工具和客户端在运行时提供的工具。前端工具通过 `RunAgentInput.tools` 传入，使用 JSON Schema 描述参数。Agent 决定调用后，通过工具调用事件发送名称和参数，前端执行对应动作，再把结果作为工具消息交还给 Agent。

这让 Agent 可以调用只存在于当前应用中的能力，例如更新筛选条件、打开指定面板、请求用户选择、读取当前选中项或把生成内容写入编辑器。Agent 负责判断何时需要该能力，前端负责将调用映射到真实界面行为。

前端工具不应被理解为把所有后端工具迁移到浏览器。数据库查询、外部 API 和服务端业务操作仍可留在 Agent 后端。只有依赖当前界面、用户上下文或本地交互的能力，才适合由客户端提供。

## Boundaries｜与相邻概念的区别 🧭

| 概念 | 主要关注的问题 | 与 AG-UI 的关系 |
|---|---|---|
| AG-UI | Agent 的运行、消息、工具和状态如何在后端与用户界面之间流动 | 提供交互事件和输入契约 |
| Generative UI | AI 结果应该渲染成什么组件，用户如何继续操作 | 可以通过 AG-UI 传递所需数据，但组件设计不是 AG-UI 本身 |
| Human-in-the-loop | 工作流在哪里暂停，哪些决策必须由人完成 | AG-UI 可以表达中断、输入和恢复，但不制定介入策略 |
| A2A | 不同 Agent 如何发现、委派任务和交换结果 | 处理 Agent 与 Agent 的协作；AG-UI 处理 Agent 与用户界面的连接 |

Generative UI 关注结果组件化，例如把查询结果渲染成卡片、表格或表单；AG-UI 关注这些结果及其运行状态如何到达前端。一个产品可以使用 AG-UI 传输事件，同时继续使用固定组件，也可以在其上实现 Generative UI。

Human-in-the-loop 是工作流控制方法。它回答“何时必须让人判断”；AG-UI 回答“暂停、输入和恢复如何在前后端之间表达”。A2A 则位于另一层，即使后台由多个 Agent 协作，对用户可见的运行过程仍然可以通过 AG-UI 进入前端。

## Product Insight｜产品观察 🔍

AG-UI 最重要的启发，是 Agent 产品的前端模型不能继续只围绕聊天消息设计。

当界面只认识 user message 和 assistant message 时，工具调用、状态更新和等待输入只能被塞进文本气泡。引入运行、步骤、工具、活动和共享状态后，产品可以把聊天保留为交流入口，同时建立任务进度、结果工作区和上下文操作界面。

协议也让前后端分工更清楚。Agent 后端负责推理、编排和产生事件，前端负责把事件转译成稳定的用户体验。两边围绕事件类型和状态 Schema 协作，而不是依赖某个框架的内部日志格式。

不过，采用 AG-UI 并不会自动带来可靠、安全或易用的产品。协议能够表达运行错误，不等于系统已经具备恢复策略；能够传递工具调用，不等于工具行为已经经过合理约束；能够同步状态，也不等于并发冲突已经解决。官方列出的集成状态只能说明相应实现提供了连接路径，不能直接推导其采用规模或生产成熟度。

## Reusable Value｜可复用的产品设计方法 🚀

### Event-to-View Mapping｜事件到界面的映射

在接入 Agent 前，先定义每类事件对应什么界面状态。运行开始后显示什么，步骤切换如何表达，工具参数是否需要展示，错误后允许重试还是重新开始，都应成为明确规则。

### Shared State Schema｜共享状态模型

把共享状态设计成稳定的产品数据模型，并区分消息、任务状态和临时活动。为 Snapshot、Delta、断线重连和状态冲突建立处理方式，避免把同步问题留给单个组件临时解决。

### Frontend Tool Registry｜前端工具注册表

为可由 Agent 调用的界面能力定义名称、参数 Schema、执行结果和失败状态。工具数量应从高价值交互开始扩展，而不是把所有按钮都包装成 Agent 工具。

### Interaction Evaluation｜交互过程评估

除了评估最终回答，还应观察首次有效反馈时间、运行状态是否容易理解、中断后能否顺利继续、状态是否保持一致，以及用户能否判断当前下一步。

AG-UI 的价值不在于给 Agent 增加一个新的展示层，而在于把 Agent 的运行过程变成前端可以理解和承接的产品状态。当事件、状态和用户动作形成闭环后，用户界面才不只是等待 Agent 返回结果的窗口，而会成为人与 Agent 共同推进任务的工作空间。
