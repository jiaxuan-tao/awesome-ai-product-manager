# Generative UI｜让 AI 输出从文本回答变成可操作界面 🎛️

- Source｜来源：Vercel AI SDK Docs
- Type｜类型：AI 产品工作流观察 / Generative UI / Tool Calling / AI-native Interface
- Link｜链接：https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces

很多 AI 产品仍然沿用最熟悉的交互方式：用户输入一句话，模型返回一段文字。

这种形式适合问答和内容生成，但当用户要查看天气、比较方案、确认订单、分析数据或继续执行任务时，纯文本会把大量理解和操作成本留给用户。

用户可能需要从回答中寻找关键数字，把选项复制到其他页面，或者重新输入一句话来表达“选择第二个方案”。模型虽然已经获得结构化结果，产品却仍然把它压缩成一段不可操作的文本。

Generative UI 提供了另一种思路：模型根据上下文决定调用什么工具，工具返回结构化数据，产品再把结果映射到适合当前任务的界面组件。

从产品视角看，它不是让模型随意生成一套网页，而是让 AI 的任务结果从“回答”升级为“下一步可以直接使用的界面状态”。

---

## What It Is｜它是什么 🧩

Generative UI 可以理解为由 AI 任务状态驱动的动态界面呈现。

根据 Vercel AI SDK 文档，一个典型流程是：产品向模型提供对话上下文和可用工具，模型判断是否调用工具，工具执行并返回数据，前端根据工具类型与执行结果渲染对应组件。

例如，天气工具返回地点、温度和天气状态后，产品可以展示天气卡片；查询订单后，可以展示订单进度和可执行操作；分析数据后，可以展示表格、图表和筛选入口。

这里的“生成”并不一定意味着模型实时编写任意界面代码。更稳健的产品方式通常是预先准备经过设计和测试的组件，再让模型或工作流决定何时使用哪个组件、传入什么数据。

因此，Generative UI 更接近“动态选择与组合界面”，而不是把产品体验完全交给模型自由发挥。

---

## Why It Matters｜为什么值得关注 💡

AI 产品的输出具有动态性，同一个输入框背后可能连接完全不同的任务。

用户询问知识时，需要看到引用与来源；比较商品时，需要看到对比维度；处理日程时，需要看到日期、参与人和确认按钮；执行 Agent 任务时，则需要看到步骤、状态和需要人工决定的节点。

如果所有结果都被渲染成相同的聊天气泡，用户必须自己从文字中重建任务结构。这不仅降低效率，也容易让重要状态被忽略。

Generative UI 的价值在于，让界面承担一部分理解和决策工作。

结构化组件可以突出关键数据，限制可选范围，提供直接操作，并把系统状态放在用户看得见的位置。用户不必每次重新组织 Prompt，而可以通过选择、编辑、确认和继续等动作推进任务。

这意味着 AI 产品的核心交互不再只是“说什么”，还包括“此刻应该出现什么界面，帮助用户完成下一步”。

---

## How It Works｜它如何工作 ⚙️

从产品工作流看，Generative UI 可以拆成意图与工具、结果与组件、状态与后续动作三个连接。

### Intent and Tool｜从意图进入工具

用户首先表达目标，模型根据上下文判断是否需要调用工具。

这一阶段决定接下来展示的是普通文本，还是某种任务组件。产品需要为每个工具定义清楚的使用条件和输入结构，避免模型在不相关场景中调用错误能力。

界面也不应因为工具调用而突然失去上下文。用户需要知道系统正在查询、计算还是执行，以及当前结果对应的是自己提出的哪个目标。

### Result and Component｜从结果进入组件

工具返回数据后，产品根据工具类型选择预定义组件。

组件的职责不是装饰回答，而是把数据变成更容易理解和操作的产品状态。同一份结果可以根据任务目标采用不同表达：少量关键数据适合卡片，多项比较适合表格，趋势适合图表，需要选择时则适合列表或表单。

这种映射需要稳定的数据契约。如果工具返回字段频繁变化，组件就会出现缺失、错误或无法渲染。因此，Generative UI 依赖的不只是模型能力，也依赖工具 Schema、输出校验和组件边界。

### State and Action｜从展示进入下一步

工具调用不是只有“成功”一种状态。

产品需要处理输入已生成但尚未执行、工具正在运行、结果已经返回、执行失败以及需要用户确认等过程。Vercel AI SDK 的示例也会根据工具消息状态分别展示加载、结果或错误。

更重要的是，组件应帮助用户继续任务。用户可以修改工具输入、重新执行、选择一个结果、确认高风险动作，或把组件中的信息带入下一轮对话。

当界面只能展示而不能继续操作时，它仍然只是更漂亮的回答；当组件能够承接下一步，Generative UI 才真正成为工作流的一部分。

## Use Case｜适用场景 📌

Generative UI 适合结果具有明确结构，并且用户需要理解、比较或继续操作的场景。

在研究与信息查询中，产品可以将来源、摘要和关键差异组织成可展开的结果卡片，让用户选择继续深入的方向，而不是面对一整段混合文本。

在数据分析中，模型可以解释问题并调用查询工具，界面则展示指标、图表和筛选条件。用户修改时间范围或分类后，系统可以直接重新计算，不必重新描述完整需求。

在 Agent 工作台中，动态组件可以展示执行计划、工具结果、人工确认点和最终交付物。不同步骤使用不同界面，但仍然保留在同一任务上下文中。

在交易、预订和表单场景中，Generative UI 还可以把模型理解转成可核对的结构。用户先检查对象、数量和时间，再明确提交，减少自然语言歧义直接进入真实操作的风险。

并不是所有回答都需要组件。简单解释、开放讨论和情绪表达仍然适合自然语言。动态界面的价值来自任务需要，而不是让每一条消息看起来更复杂。

## Product Insight｜产品观察 🔍

Generative UI 最重要的产品启发是：AI 产品不必在“固定页面”和“空白聊天框”之间二选一。

固定页面清楚稳定，但难以覆盖用户不断变化的意图；聊天框灵活，却容易把结构、状态和操作全部隐藏在文字中。Generative UI 尝试把两者结合：用户用自然语言表达目标，产品根据任务动态调用经过设计的界面能力。

它与仓库中已有的 PromptInfuser 有明确区别。PromptInfuser 关注设计阶段如何让 Prompt 输出和 UI 原型共同验证；Generative UI 关注产品运行后，工具调用结果如何实时进入界面，并继续承接用户操作。

它也不等同于 Tool Calling UX。Tool Calling UX 关注用户如何理解工具正在做什么、如何确认和追踪；Generative UI 更关注执行结果最终以什么形式出现，以及用户如何通过这个结果继续任务。

另一个关键判断是，界面生成不应破坏产品一致性。如果每次模型都生成完全不同的组件和交互规则，用户会难以预测，也难以建立操作习惯。

真正成熟的方式，是让模型在受控的组件系统中选择和组合。AI 负责理解任务，设计系统负责保持一致性，工具契约负责保证数据可靠。

## Reusable Value｜可复用价值 🔁

设计 Generative UI 时，可以先从高频任务结果出发，而不是先追求“任何界面都能生成”。

团队需要判断哪些输出具有稳定结构，哪些结果需要用户进一步操作，以及哪些状态用纯文本表达会增加理解成本。随后为这些场景建立少量清晰组件，并定义它们所依赖的数据字段和异常处理。

每个组件还应回答三个问题：用户现在看到的是什么，这个结果来自哪个工具，接下来可以做什么。

如果工具失败或数据不完整，产品需要提供文本回退、重试或人工输入，而不是让组件空白。对于高风险动作，动态界面也必须保留明确确认，不能因为模型已经理解意图就自动执行。

这套方法可以用于 AI 搜索、数据助手、知识库、Agent 工作台和 Vibe Coding 项目，帮助它们从单一聊天界面走向更清楚的任务体验。

## Productization Direction｜产品化方向 🚀

### AI Component Registry｜AI 组件注册表

产品可以建立工具与组件的对应关系，记录每个组件需要什么数据、适合什么任务、有哪些加载和错误状态。模型只能选择经过验证的组件，降低不可预测性。

### Interactive Result Workspace｜交互式结果工作区

AI 输出可以从聊天消息中进入一个可持续编辑的工作区。用户在其中比较、筛选、修改和确认，聊天则保留解释与协作能力。

### Generative UI Evaluation｜动态界面评估

团队不仅要评估模型回答是否正确，也要评估组件是否选对、数据是否完整、用户是否顺利完成下一步，以及文本回退是否有效。

Generative UI 展示了 AI 产品从“生成内容”走向“生成任务体验”的方向。

它真正改变的不是页面是否更丰富，而是用户获得结果后，不再需要自己把文字转成行动。产品可以用受控、稳定的界面组件，把 AI 的理解直接连接到下一步任务。
